## 二、筛选器重做:as-of 账期 + 窗口 + MoM/YoY

### 决策 99 · 用 `rangeEnd = asOfPeriod` 解除「永远最新期」的硬编码

**现状**:`FactFilter(familyId, periodType, rangeStart, rangeEnd, includeArchived, accountIds, viewCurrency)`,`rangeEnd` 在 `DashboardController:87-95` 永远锁死 `anchor.getPeriodStart()`(最新期)。所有点状 KPI 取 `slice.lastPeriodId()`,于是与 range 无关 —— 这是 Q2 的病根。

- **选定**:新增**观察账期 as-of**(默认最新,可选历史月),把它喂给 `rangeEnd`;窗口(近 N 期 / YTD / 全部)只决定 `rangeStart`。于是:
  - 点状 KPI(净资产/总资产/负债/配置/账户当前值)取 `slice.lastPeriodId()` —— 因 `lastPeriodId == rangeEnd == asOf`,**自动变成「那个月」的状态**,零额外改动。
  - 窗口型(趋势图/区间 XIRR/TWR/净资产变化)用整个 slice(`rangeStart..asOf`)。
  - 趋势图 x 轴 = 窗口;终点 = as-of(不再永远到最新)。
  - 修掉 1M Δ=「—」:Δ 改由 MoM 提供(决策 100),与窗口大小无关。
- **备选(否)给 FactFilter 新增独立 `asOfPeriodId` 字段**,与 rangeEnd 并存:语义更直白,但要改 FactMapper 多处 + 所有取 lastPeriodId 的地方判该用谁。`rangeEnd=asOf` 复用现有 BETWEEN,改动面最小、零 SQL 改。
- **备选(否)维持单一 range,只把更多 KPI 接上**:选 3M 时「净资产显哪期」无解,语义仍别扭(PRD 决策 91 已否)。
- **UI 控件**:as-of = 月份步进/下拉;窗口 = segmented(近3/6/12期·YTD·全部)。URL `?asof=YYYY-MM&window=6M&currency=&accounts=`(承现有 `?range=`,旧链接 `range` 做窗口别名兼容)。

### 决策 100 · MoM/YoY = 对比账期的**独立最小加载**,不受显示窗口约束,不落库

**难点**:MoM 要 asOf−1 期、YoY 要 asOf−12 期的数据,可能落在显示窗口之外。

- **选定**:MoM/YoY 由一次**覆盖 `[asOf−12, asOf]` 的最小 slice**(或对两个对比期的点查询)算出,**与显示窗口解耦**。家庭级取该期 `netWorth`,账户级取该期 `endBalanceBase`;算 Δ + %;**对比期不存在(数据不足 1 年 / 缺月)→ 返回 null,UI 显「数据不足」**,不臆造。实时算(决策 92)。
- **备选(否)把显示窗口强行扩到含 asOf−12**:让 MoM/YoY 蹭主 slice。会污染趋势图 x 轴(用户选近3期却被拉长),把「显示」和「计算」耦死。否。
- **备选(否)落库**:见 PRD 决策 92——账期可 reopen、派生纯函数、数据量极小,落库只增一致性负担。否。
- **成本**:一个家庭几十~百行,多查 12 行可忽略;`FactViewService.load` 已无副作用可并发。

### 二.改动清单
`FactFilter`(语义注释:rangeEnd=asOf)· `DashboardController`(收 asof/window 两参 + 算 rangeStart/rangeEnd + MoM/YoY 注入)· `FactViewServiceImpl`(+`momYoy(slice/familyId, asOf)` 辅助,返回家庭级与各账户的环比/同比)· `_region.html`(双控件 UI + KPI 卡挂 MoM/YoY 徽标 + 趋势终点=asOf)。

## 三、指标端出 + 可配置指标集

### 决策 101 · 扩 `AccountPerformance`,在 `accountPerformance()` 一趟算全集(不要 per-account diagnose)

**现状**:dashboard 列表用精简 `AccountPerformance(currentValue, xirr, sparkline)`;丰富指标在 `AccountDiagnoseService.diagnose()`,但它**每账户一次 12 月加载**,列表 N 个账户会 N 次加载。

- **选定**:`accountPerformance(slice)` 已按账户分组持有 rows;**在这同一趟里多算几个字段**——`cumPnl`(ΣperiodPnl)、`netPrincipal`(Σ净外部流入+净转账)、`periodDelta`+MoM%、`sharePct`(账户现值÷家庭净资产)、`maxDrawdown`(NavSeriesBuilder+MaxDrawdown)、`benchmark/预实`(BenchmarkComparator)、`monthsHeld`、`risk`、`sparklinePoints`(渲染好的 SVG points,见决策 103)、`returnOrig`/`returnBase`(决策 102C)。复用现成 calc 纯函数。`AccountPerformance` record 加这些字段。
- **备选(否)列表里对每账户调 `diagnose()`**:N 次独立 slice 加载,浪费;且 diagnose 固定 12 月窗口,与 dashboard 的 as-of/窗口口径不一致。否。
- **备选(否)把 AccountDiagnose 直接塞进列表**:它含 account/category 等重对象,序列化进 JS 偏重;精简扩展版更合列表。
- **一致性**:体检页继续用 `diagnose()`(全集 + 归档可查);列表用扩展版 `accountPerformance()`——两者复用同一批 calc 函数,口径一致。

### 决策 102 · 指标勾选配置存 `family.metric_prefs` JSON

- **选定**:`ALTER TABLE family ADD COLUMN metric_prefs JSON NULL`。结构如 `{"family":["net_worth","family_xirr",...],"account":["current_value","xirr","cum_pnl",...]}`。NULL → 用代码默认集(FR-150)。读取走现有「DB→默认」fallback 心智。
  - 仿 `family.allocation_anchor_custom JSON` 既有先例;一行、无 join、改一处。
- **备选(否)新表 `family_metric_config(family_id, scope, metric_key, enabled, ord)`**:规范、可排序,但要建表 + CRUD + join,对「几十个布尔勾选」过重。留作「将来要 per-指标自定义顺序/per-页覆盖」时再升。
- **备选(否)塞 `family_runtime_config`(V26 kv)**:那是运营参数(aksk/阈值/cron)语义,指标显示偏好是另一类,不混。
- **必选项**(当前价值/净资产)在**渲染层**强制纳入,即便配置漏了也显;进阶项默认不在 NULL 默认集里。
- **上限裁剪(决策 96)**:在渲染层按「目录优先级顺序」对 enabled 集取前 N(手机卡片 4 / 桌面表 6 / KPI 4-6 / reports 不限)。优先级 = 代码内固定的 metric 目录顺序。

### 决策 103 · sparkline = 后端生成归一化 SVG points,Thymeleaf 内联(替换硬编码假图)

- **选定**:`accountPerformance()` 把 `List<TrendPoint>`(月末余额序列)线性映射到 `viewBox 0 0 80 22` 的 points 字符串 + 涨跌方向,作为 `sparklinePoints`/`sparklineTrend` 字段;`_region.html` `th:attr="points=..."` 渲染,`th:classappend` 涨绿/跌红;**点数 <2 → 不画线,显「—」或单期数值**(decision in template `th:if`)。
- **备选(否)纯前端 JS 读 `account.sparkline` 画**:DOM 顺序匹配 fragile(探查已指出);后端生成 points 更稳、SSR 友好。
- 受窗口/币种影响(用 slice 内该账户的 endBalanceBase 序列)。

## 四、计算正确性

### 决策 104 · 跨币种转账 `to_amount`(承 PRD 决策 93)

`V30__transfer_to_amount.sql`:`ALTER TABLE transfer ADD COLUMN to_amount DECIMAL(18,2) NULL`(NULL=同币种)。`FactMapper.xml` 转入子查询 `SUM(amount)`→`SUM(COALESCE(to_amount, amount))`;转出仍 `SUM(amount)`(转出账户本币)。`Transfer` 域 + `TransferMapper` insert/update 加该列;转账表单**跨币种时显「到账金额」**(按 fx 预填、可改)。计算逻辑(`xirrForAccountRows`/`PnlCalculator`)零改。旧记录 NULL→COALESCE 回落,零影响。

### 决策 105 · Problem B:手动调股票内部现金行 → 记一笔「调整」外部流,剔出 PnL

**现状**:`StockHoldingService.updateCashAmount` 改 CASH 持仓 manual_value → endBalance 变 → 被恒等式当 PnL(污染 XIRR)。

- **选定**:`updateCashAmount` 把现金从 X 改到 Y 时,**对该账户当期记一笔 `cash_flow` 调整**(金额 = Y−X,正记 INCOME 向、负记 EXPENSE 向),使恒等式把它从 PnL 剔除(当作本金进出,不是投资收益)。为不污染「储蓄率/月均收支」统计,该流水打**调整标记**:
  - 方案落点(tech-design 内定):cash_flow 加一个可空 `kind_detail`/`is_adjustment` 标记;`FactMapper` 的 income/expense 仍计它(因为账户级净外部流入要含它),但**家庭级储蓄率/收支聚合(PMC 路径)排除调整流**。
- **备选(否)在估值层把 CASH 行变动单独标为 flow**:要改估值写回链 + 多记一张事件表,重。复用 cash_flow 最小。
- **备选(否)不修,文档说明「别手动改现金行、改用转账/收支」**:治标,用户照样会手滑。选定方案从根上把它归对类。
- **注意**:AUTO 持仓的估值变动**不**触发本逻辑(那才是真投资损益);只针对 CASH 行的**手动**改。

### 决策 106 · Problem C:非本位币账户给「原币收益率」+「本位币收益率」双值

**现状**:账户级 XIRR(`xirrForAccountRows`)用 **orig** 币种 → 即「原币收益率」,**不含 FX**;家庭级用 base → 含 FX。FX 损益现混在本位币净资产变化里、未单列。

- **选定**:对**非本位币账户**同时给两个收益率:
  - **原币收益率**(现有 orig XIRR,剔除汇率,看标的本身);
  - **本位币收益率**(新增:account XIRR 用 base 现金流序列,含 FX,对家庭的真实回报)。
  本位币账户两者相等 → 只显一个。家庭级 PnL/收益率**保持含 FX**(对家庭是真实损益),不强拆。
- **备选(否)单列一条「汇率损益」金额**:更细,但要在 PnL 里再切一刀(asset-return vs fx-return),改动大、易错;双收益率已让用户看清「标的 vs 含汇率」差异,够用。
- **实现**:`accountPerformance()` 各账户多算一个 base 口径 XIRR(rows 已有 endBalanceBase/incomeBase/transferInBase)。零 schema。

## 五、列表排序(决策 94)

- **选定**:纯前端 JS。每张可排序表 `<th data-sort="num|text" data-key="...">`,每行 `<td data-val="...">` 带可排序原始值(数值列放未格式化数字,空值给 `+Infinity` 之类保证沉底)。点击 th 在 `none→asc→desc→none` 循环,JS 重排 `<tr>`,写 `aria-sort` + ▲▼ 图标。HTMX 局部刷新后重新绑定(脚本幂等)。
- **备选(否)服务端 `?sort=`**:数据已在 DOM 且量小,无需往返/新 endpoint;留作将来分页。
- 覆盖:dashboard 账户表、`/accounts` 账户簿(P1);reports 表(分期可选)。

## 六、预实分析(决策 98)

- **数据**:`V31__account_expected_return.sql`:`ALTER TABLE account ADD COLUMN expected_return_pct DECIMAL(5,2) NULL`(NULL→回落品类 `product_category.benchmark_pct`)。账户编辑表单加可选「预期年化%」。
- **计算**:复用 `BenchmarkComparator.compare(actual, expected)` —— 把「基准」语义改叫「预期」,**合并**基准对照与预实为一个展示组件。
  - 收益率维度:实际(年化 XIRR / 区间)vs 预期% → 差异 bps + 达成率(actual/expected)+ 跑赢/跑输。
  - 金额维度:预期收益金额 ≈ 平均投入 × 预期% × 持有年数;实际 = `cumPnl`;差额。
- **缺失态**:无 expected 也无品类 benchmark → 「未设预期」;持有 <2 期 → 「数据不足」。
- 作为账户级可勾指标(默认 ✓),列表/卡片显紧凑「实际 vs 预期 + 达成率徽标」,详情/体检看金额维度。

## 七、验收 / 红线

- **真机口径验证**(beta 隔离测试库或构造数据):跨币种转账两端算对 + 家庭净零;手动改股票现金行后该账户 XIRR 不被污染(调整流被剔除);USD 账户 FX 波动时「原币收益率」稳、「本位币收益率」含 FX;MoM/YoY 缺对比期显「数据不足」;as-of 选历史月时点状 KPI 随之变、窗口独立控趋势。
- **不回归既有数值**:同币种、本位币账户、默认窗口下,净资产/XIRR/TWR/PnL 数值与 v0.7 一致(新增字段不改旧口径)。
- **schema 红线**:`transfer.to_amount` / `account.expected_return_pct` / `family.metric_prefs` / Problem B 的 cash_flow 标记,全为新增可空,4 条 schema 自检 + 部署红线(承 [[feedback_prod_backward_compat]])。
- 派生指标不落库(决策 92);改 `_region.html` 等长文目录页同步 TOC(承 [[feedback_toc_sync]]);qa-run 新增 `v08-*` 守护;mvn test 全绿。
- **分期**:P1(A 渲染+D 排序)→ P2(E 配置+预实)→ P3(C 跨币种+Problem B/C 正确性)→ P4(B 筛选器)。每期可独立发 + 真机验。
