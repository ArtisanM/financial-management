## 0. 架构选型一览

| 项 | 选择 | 备选 | 选定理由 |
|---|---|---|---|
| 财务目标存储 | 单表 family_goal + params_json | 多张子类型表(family_goal_retirement / _education / _emergency) | 三类目标参数差异大但都简单(< 8 字段),JSON 字段灵活 + forward compat;子表方案需 3 个 Mapper + 3 套 SQL,过度工程 |
| 预测模型 | 三情景复利公式(纯函数) | 蒙特卡洛 / 历史回放 | 简洁 · 可视化直观 · 妻子能看懂 · 用户已拍 |
| 目标进度口径 | 全资产(净金融 + 房产净值)· 应急例外 | 账户绑定 / 类型映射 / 专项标记 | 零设置 · 用户主动选 |
| 股票数据源 | 新浪(主)+ 腾讯(备) | Yahoo / AlphaVantage / 付费 | 国内可达 · 免费 · 无 key · 三市场统一接口 |
| 持仓估值模式 | 持仓级 AUTO/MANUAL 混合 | 账户级 boolean 开关 | 用户提需求:字节跳动期权这类未上市的必须能 MANUAL,而同账户的 BABA 又能 AUTO |
| 自动估值的下游集成 | 写回 account_balance 表 | 新建并行表 + fact_view 改写 | account_balance 是 v0.2 已有抽象 · 写回它意味着 dashboard / XIRR / 目标进度 全部下游零改动 |
| 周期月度家庭口径 | period 加 2 NULL 列 | 新建 period_summary 表 | NULL 安全 · 一次 JOIN 解决 · 不引入新表关系 |
| 拉价 cron | Spring @Scheduled · 3 个固定 cron | Quartz / Spring Batch | 单实例部署 · 不需要分布式调度 · 复杂度低 |
| AI 介入 | 复用 v0.2 LlmOrchestrator + OutputValidator | 各 FR 独立 prompt 框架 | 真名脱敏 / 担保性词拒绝 / 圆形熔断 全部 v0.2 已有 |
| AI 节流 | 90 天/目标(预警)+ 7 天/周期(月报)+ 实时(向导) | 全实时 / 全异步 | 月报 / 预警是消费侧异步 · 向导是用户触发实时 |
| 信息架构 | C 混合(dashboard 条带 + /goals 一级页) | A 独立 /goals / B dashboard 改造 | 用户拍板:目标可见但净资产不被降级 |

## 2. 关键决策详解

### 决策 21 · 目标存储用单表 + JSON 还是分表

**背景**:3 类目标参数差异大,RETIREMENT 5 个字段,EDUCATION 6 个字段(含 child_member_id FK),EMERGENCY 2 个字段。

**备选**:
- A. 单表 `family_goal` + `params_json JSON NOT NULL`(选)
- B. 主表 `family_goal` + 3 张子表 `family_goal_retirement` / `_education` / `_emergency`(继承表模式)
- C. 一张大表所有字段都列出,按 type 决定哪些 NULL

**取舍**:
- A:1 张表 · 1 个 Mapper · 加新目标类型不动 schema · JSON 字段索引差但查询全量低频 → 选
- B:类型完整 · 但每加一个目标类型加一张表 + Mapper + xml · forward compat 差
- C:NULL 字段污染严重 · 用户看 schema 困惑 · 没有约束

**选 A 的额外保障**:
- params_json 在 Service 层用 Jackson 反序列化到类型化 record(GoalRetirementParams / GoalEducationParams / GoalEmergencyParams)
- 数据库不做 JSON Schema 约束(MySQL 8 支持但跨厂商性差)· 由 Service 层 validator 兜底

### 决策 22 · 预测模型选三情景而非蒙特卡洛

**备选**:
- A. 单点预测(用户输入 1 个 r)
- B. 三情景(乐观 8% / 中性 5% / 悲观 2%)三曲线(选)
- C. 蒙特卡洛(分布抽样 N=1000)
- D. 历史回放(假设未来重演)

**取舍**:
- A:简洁但隐瞒不确定性 · 用户会以为是确定预测
- B:**3 条线 + 3 个达成日** · 用户一眼看到"乐观/悲观区间" · 数学复杂度可控 → 选
- C:专业但参数多(σ 怎么定?)· 可视化"概率密度图"对非金融背景用户陌生 · 解释成本高
- D:依赖历史数据足够长 · 对应"过去 N 年" · 但首次用户没有数据;且过去未必重演

用户在 brainstorming 时拍板 B。

**预设值的来源**:
- 8% / 5% / 2% 是 A 股 / 美股长期年化的"宽参考"(沪深 300 近 20 年 ~8%,标普 500 近 30 年 ~7%,通胀 ~2%)
- 不允许用户改三情景的 r(避免参数过载)· 但 v0.4 可以加"自定义"模式

### 决策 23 · 目标进度口径选"全资产"而非"账户绑定"

**备选**:
- A. 全资产(净金融 + 房产净值)(选)
- B. 用户勾选账户(多对多)
- C. 按账户类型预设映射(退休=STOCK+WEALTH+PROPERTY · 教育金=STOCK+WEALTH 等)
- D. 账户加"专项标记"字段(退休/教育/应急/未指定)

**取舍**:
- A:零设置 · 视角"家庭整体资产支撑所有长期目标" · 现实中钱本来就是流动的 → 选
- B:精确 · 但首次创建目标要勾 N 个账户 · 违反"快速创建"
- C:粗暴 · 房产不一定是退休资产 · 强映射会误判
- D:类似 B 但更死板 · 一个账户只能服务一个目标

**A 方案的副作用**(用户接受):
- 退休进度可能初始 80%+(中年家庭金融资产 + 房产已接近退休金)
- 多个目标会共享同一笔资产(教育金和退休都算同一笔股票)
- 现实视角下这是对的:家庭资产是一个池子,不是 N 个隔离账户

**例外**:应急储备进度只算 CASH 类(流动性硬约束) · 这是该目标的语义本质,不应用全资产口径。

### 决策 24 · 股票数据源选新浪 + 腾讯而非 Yahoo

**备选**:
- A. 新浪(主)+ 腾讯(备)(选)
- B. Yahoo Finance(主)+ Alpha Vantage(备)
- C. 付费 API(Polygon / Wind / 同花顺)

**取舍**:
- A:免费 · 无 key · 国内服务器直接可达 · 一个源覆盖三市场 · unofficial 但 10+ 年稳定 → 选
- B:Yahoo 在国内访问不稳定(域名间歇被墙)· Alpha Vantage 5 req/min quota · 都需绕路 / 配代理
- C:付费 · 月费几十至几百美元起 · 违反"自托管 + 家庭使用"的成本约束

**实施风险与缓解**:
- 接口随时可能改格式 → 加解析异常 audit log + 健康检查 endpoint
- charset 是 GB18030(美股/A股)/ UTF-8(港股某些字段)· RestTemplate 配 StringHttpMessageConverter 明确指定
- 需 `Referer: https://finance.sina.com.cn` 头(新浪 403 防护)

### 决策 25 · 持仓估值模式做"持仓级 AUTO/MANUAL"

**备选**:
- A. 持仓级 AUTO/MANUAL 字段(选)
- B. 账户级 boolean 开关
- C. 全自动(账户类型=STOCK → 必须录持仓 · 强制 AUTO)

**取舍**:
- A:一个账户内混合 BABA(AUTO) + 字节期权(MANUAL) → 选
- B:用户场景"招商证券港美股账户里既有 BABA 又有字节跳动期权"被堵死
- C:字节期权 / 私募基金 / 港股仙股(新浪查不到) 完全无解

**用户原话**(2026-05-12):"如果用户托管的股票 你无法拉取到市值(失败,或者类似字节跳动期权 本没有上市)这种情况 依然允许用户月度手动更新余额"。

### 决策 26 · 自动估值的输出落点

**备选**:
- A. 写回 `account_balance` 表(下游不变)(选)
- B. 新建 `stock_holding_valuation` 表,fact_view 改写为 UNION(account_balance, stock_holding_valuation)
- C. 实时算 · 不持久化 · dashboard 每次查询都 SUM 持仓

**取舍**:
- A:**v0.2 所有下游(fact_view / XIRR / dashboard / 目标进度 / checkup)零改动** · 历史 account_balance 不变 · 自动估值定时跑写新行 → 选
- B:fact_view 复杂度 ↑ · v0.2 现有指标全部要 review · 风险大
- C:实时算无缓存 · 接口慢 · 历史数据无法追溯

**实施细节**:
- 自动估值 cron 跑完 → 对每个 STOCK 账户算 currentValue → `account_balance.upsert(periodId=OPEN, accountId, balance, source='AUTO_STOCK')`
- 加 `source` 字段区分人工填 / 自动算(audit 用)→ 实际不加,避免 schema 改动,通过 audit_log 表记录每次 AUTO 写入

### 决策 27 · 周期月度家庭口径用 +2 列还是新表

**备选**:
- A. `period` 加 2 NULL 列 `total_income_input` / `total_expense_input`(选)
- B. 新建 `period_summary` 表(family_id, period_id, total_income, total_expense)
- C. 加 `family_cashflow_aggregate` 表(每月一行)

**取舍**:
- A:周期级数据天然属于 period · 一次 JOIN(实际不需要 JOIN,直接 SELECT)· NULL 安全 → 选
- B:多一张表 · 多一个 Mapper · 数据本身就属于 period · 没必要分
- C:本质同 B

### 决策 28 · AI 月报触发机制

**备选**:
- A. 周期关闭 hook 异步生成 + 缓存(选)
- B. 用户访问 /goals/{id} 时实时生成
- C. cron 每月跑一次

**取舍**:
- A:周期关闭是天然事件 · 异步不阻塞 PeriodCloseService 返回 · 缓存到下次周期关 → 选
- B:每次访问触发 LLM 调用 · 成本高 · 慢(2-6s)· 同一周期多次刷新报告内容变
- C:cron 时机难定(每月几号?)· 周期关闭时机本身就准确,不需要再 cron

**实施细节**:
- PeriodCloseService 关周期成功后 · `taskExecutor.submit(() -> goalReportService.generateForAllGoals(familyId, periodId))`
- 月报存 `goal_ai_report(goal_id, period_id, content, generated_at)` 表 · v0.3 不建表,而是塞到 `family_goal.params_json` 末尾(`_latest_report` 字段),省一张表
  - **修正**:此项 brainstorming 阶段未充分讨论,实施 review 时定;首版可塞到 audit_log 表 + 一个新的简单表 `goal_ai_report`

(决策 28 标记为"实施前再 review"· 周期月报的持久化方式有 2 种实现,v0.3 阶段先用简单实现:`goal_ai_report` 表)

### 决策 29 · 偏离预警节流策略

**备选**:
- A. 同目标 90 天内最多 1 次(选)
- B. 每月 1 次
- C. 实时 · 满足条件就触发

**取舍**:

### 决策 32 · 股票账户内现金的建模(2026-05-13 追加 · FR-52e)

**问题背景**:用户反馈,STOCK 账户既要支持多个股票/多币种持仓(决策 25 已覆盖),**还要能录入账户内的闲置现金**(IBKR / 富途 / 老虎 / 国内券商账户里没买股票的那部分钱),且现金本身也是多币种的(港币现金 + 美元现金共存)。

**问题**:`AccountValuationService.refreshAllForFamily` 检测到账户有任何 stock_holding 行就用 SUM(持仓市值) 覆盖 `period_snapshot.end_balance`,用户在 /entry 填的余额(可能包含现金)被冲掉,导致 STOCK 账户的现金部分**完全无处可填**。

**备选**:
- A. `stock_holding` 表加第三个 ValuationMode `CASH` · 现金作为"持仓行"的一种 → **选**
- B. 新建 `stock_account_cash` 表,字段 (account_id, currency, amount, updated_at)
- C. 账户层加 cash_balance + cash_currency 列(只支持单币种)
- D. 退回 /entry 手填语义:STOCK 账户保留 v0.2 手填行为不接管(但失去 AUTO 自动估值的价值)

**取舍**:
- A:语义对仗清晰(AUTO/MANUAL/CASH 都是"账户里有什么"的一行),不动 schema(只多枚举值),`stock_holding.findActiveByAccount` 一次查询拿全部,估值逻辑加一个分支即可 → 选
- B:多一张表 + 多一个 Mapper + UI 要单独展示区,但本质上跟 MANUAL 持仓没区别(都是用户手填一个币种+金额),分表是过度设计
- C:只支持单币种,跟决策 25 多币种混合冲突 · 否决
- D:用户体验差(自动估值价值消失);且 v0.2 手填的是单一余额,无法表达"多币种现金分别多少"

**实施**:
- `ValuationMode` 枚举加 `CASH` 第三类
- `stock_holding` 字段语义复用矩阵(见 ValuationMode javadoc):
  - AUTO:ticker+market+shares+currency 必填,manual_value NULL
  - MANUAL:manual_value 必填(账户币种值,不做 FX),其它 NULL
  - CASH:currency+manual_value+manual_value_at 必填(currency 是现金币种,manual_value 是该币种金额,**估值时 FX 至账户币种**),其它 NULL
- `AccountValuationService.valuateInternal` switch 三分支:
  ```
  total = SUM(AUTO × FX) + SUM(MANUAL 直填) + SUM(CASH × FX(CASH.currency → account.currency))
  ```
- `ValuationResult` 加 `cashBaseValue` 字段(record 加一个组件)
- UI:`/accounts/{id}/holdings` 操作栏加"+ 账户内现金"按钮 + 新页 `/holdings/new-cash`(币种下拉 + 金额输入)
- Schema:**不需要 V20 迁移**(纯枚举值扩展 · `valuation_mode` 列已是 VARCHAR(8))

**为什么不要 V20**:
- `stock_holding.valuation_mode` 是 VARCHAR(8) 不是 ENUM 类型,加新值不动 DDL
- `currency` 字段 V16 已是 NULL-able,CASH 行的 NOT NULL 语义靠 Service 层 validate
- `manual_value` 字段已存在 V16 · CASH 行复用
- V16 的列 COMMENT 不更新没关系(注释只是给人看的元数据,运行时不依赖)

## 4. 数据流图(关键路径)

### 4.1 目标进度计算

```
用户访问 /goals 或 dashboard
   ↓
GoalController.list(familyId)
   ↓
GoalService.findAllActive(familyId) → List<Goal>
   ↓
对每个 Goal:
   GoalProgressCalculator.compute(goal, currentFactView)
   ├─ PV = ?
   │   ├─ goal.type=RETIREMENT/EDUCATION → SUM(fv_account_balance.current_value_base WHERE type IN ('CASH','STOCK','WEALTH','PROPERTY')) - SUM(... WHERE type='LOAN')
   │   └─ goal.type=EMERGENCY → SUM(fv_account_balance.current_value_base WHERE type='CASH')
   ├─ Target = ?
   │   ├─ RETIREMENT → params.monthlyExpense × 12 × (1+inflation)^年数 / withdrawalRate
   │   ├─ EDUCATION → params.targetAmount × (1+inflation)^(targetYear - currentYear)
   │   └─ EMERGENCY → params.autoBaseline ? recentExpenseMedian × monthsTarget : fixedBaseline × monthsTarget
   └─ progress = PV / Target
   ↓
GoalProjector.project(goal, PV, monthlyContribution)
   ├─ monthlyContribution = median(过去 6 期 (total_income_input - total_expense_input)) · NULL 跳过 · 全 NULL → 0
   ├─ 对每个 r ∈ {0.08, 0.05, 0.02}:
   │   ├─ FV(n) = PV × (1+r)^n + monthlyContribution × ((1+r/12)^(12n) - 1) / (r/12)
   │   └─ 二分法解 n 让 FV(n) ≥ Target → achievementDate
   └─ 返回三情景 path[months] + achievementDates[3]
   ↓
渲染 dashboard 条带 / /goals/{id} 详情图
```

### 4.2 股票自动估值

```
@Scheduled cron 触发(每个市场单独)
   ↓
StockPriceScheduler.fetchUsStocks() / fetchCnStocks() / fetchHkStocks()
   ↓
对应市场所有 AUTO 持仓的 distinct ticker 集合
   ↓
StockPriceFetcher.fetchBatch(market, tickers, today)
   ├─ SinaStockClient.fetch(market, tickers) · 成功 → 写 stock_price_snapshot
   ├─ 失败 → CircuitBreaker.recordFailure(sina)
   ├─ Sina 短路 → TencentStockClient.fetch(market, tickers)
   ├─ 都失败 → 跳过(用历史 snapshot)
   └─ 都成功 → 优先写最新源数据
   ↓
触发 AccountValuationService.refreshAllStockAccounts()
   ├─ 找所有 STOCK 类型账户
   ├─ 对每个账户:
   │   ├─ holdings = stockHoldingMapper.findActiveByAccount(accountId)
   │   ├─ 对每个 AUTO holding: 找最新 snapshot · 算 marketValue (shares × price × fx_to_base)
   │   ├─ 对每个 MANUAL holding: 用 manual_value
   │   └─ totalBalance = SUM(autoTotal + manualTotal)
   └─ accountBalanceMapper.upsertForCurrentPeriod(accountId, totalBalance)
   ↓
下游 fact_view 自动反映(SELECT 时实时聚合)
```

## 6. 风险与缓解

### 6.1 数据源稳定性

| 风险 | 影响 | 缓解 |
|---|---|---|
| 新浪接口变更格式 | 全市场拉价失败 | 熔断切腾讯 · audit log 记录 · 监控 endpoint /admin/stock-prices · 7 天历史价兜底 |
| 新浪反爬升级(IP 封) | 同上 | 切腾讯 · 长期需多源轮换(v0.3.x 引入) |
| 港股零碎股代码歧义(0700 vs 00700) | 入库不一致 → 拉不到价 | Service 层入库前 padStart 5 位 · UI 显示 4 位 |
| GB18030 解析乱码 | display_name 乱码 | RestTemplate StringHttpMessageConverter 显式配 charset |

### 6.2 LLM 风险

| 风险 | 影响 | 缓解 |
|---|---|---|
| AI 推荐目标默认值不合理(monthly_expense=10万) | 用户被误导 | OutputValidator JSON parse + 字段范围校验 |
| 月报担保性话术("保证退休前达成") | 用户被误导 | v0.2 OutputValidator 已有规则全部触发 |
| 预警过度触发(每月扰民) | 用户烦 | 90 天节流 + dismiss 30 天不再弹 |
| AI 输出泄露孩子真名 | 隐私 | PromptBuilder 脱敏 + OutputValidator 真名检查 |

### 6.3 数据完整性

| 风险 | 影响 | 缓解 |
|---|---|---|
| AUTO 持仓拉价持续失败 | 账户余额不准 | UI 显示陈旧 N 天 · 7 天后强提醒 · 用户改 MANUAL |
| 用户全 NULL 月度收支 | 目标月供反推用 0 | UI 红条引导 · 不阻塞创建 · 曲线照画(无月供贡献) |
| 应急基线自动算需要 3 期历史 | 创建时显示 0 | 自动基线灰显 · 引导补录 · 允许手填 fixed_baseline |

### 6.4 backward compat 红线

- V14-V17 全部新增 · 不破坏 v0.2 数据
- period +2 NULL 列 · v0.2 查询 ResultMap 自动忽略
- account_balance 表不动 · 新自动估值数据走原 upsert
- 所有 v0.2 路由 / 模板不破坏(扩展不 breaking)
- LlmOrchestrator API 不变 · 新 Service 复用

### 6.5 总体回退

任意模块 prod 异常:
- 模块 A(/goals)异常 → 路由 503 · nav 「目标」可隐藏 feature flag · 其它不动
- 模块 B(/entry +2 框)异常 → 卡片隐藏 · 旧填报路径不变
- 模块 C(自动估值)异常 → cron 跳过 · 历史价 fallback · 持仓改 MANUAL
- 数据库迁移失败 → 单条 DROP COLUMN / TABLE 回退

`deploy/rollback.sh` 完整 jar 回退路径保留。

## 10. 部署 · macOS 支持(2026-05-13 追加)

> **背景**:v0.3 收尾期增加本机 mac 部署支持(本地开发 / 个人自用),无需 Linux 服务器。

### 10.1 设计选择 · 方案 C(顶部 dispatch)

3 个候选:

| 方案 | 做法 | 维护成本 | 用户体验 |
|---|---|---|---|
| A · 单文件 if/else 分支 | 在 deploy.sh 每个步骤 `if OS=darwin then ... else ... fi` | 高(主线 500+ 行被 OS 标签污染) | 一致(都 `sudo bash deploy.sh`) |
| B · 用户分情况调脚本 | Linux 用 deploy.sh · macOS 用 deploy-macos.sh | 低(各自独立) | 不一致(用户得记) |
| **C · 顶部探测 + exec 转发** | deploy.sh 顶部 `[[ $(uname) == Darwin ]] && exec bash deploy-macos.sh` | 低(2 个文件,各管各的) | 一致(用户都 `bash deploy.sh`) |

**选 C 的理由**:
- 用户记忆一致(无论平台都 `bash deploy/deploy.sh`)
- 主 deploy.sh 保持 Linux 路径不污染(原 500+ 行不动)
- macOS 分支独立小脚本(~280 行),无 sudo / systemd / nginx 概念
- exec 替换进程 · 没有"分支调用"的开销和栈污染

### 10.2 macOS 路径差异

| 项 | Linux | macOS |
|---|---|---|
| 入口 | `sudo bash deploy.sh` | `bash deploy.sh`(无 sudo) |
| 应用目录 | `/opt/finance` | `$HOME/finance` |
| env 文件 | `/etc/finance.env`(640 root:finance) | `$HOME/.finance/finance.env`(600 user) |
| 服务管理 | systemd(`finance.service`) | launchd(`finance.macos.plist.template` · 可选) |
| 启动 | `systemctl start finance` | `bash $HOME/finance/start.sh`(生成的 wrapper) |
| 日志 | `journalctl -u finance -f` | `tail -f $HOME/finance/logs/app.log` |
| 包管理 | `apt-get` / `dnf` | `brew install` |
| 反代 | nginx :80 → :SERVER_PORT | 无(直接访问 :SERVER_PORT) |
| 应用用户 | `finance` 系统用户 | 当前 macOS 用户 |
| sha256 | `sha256sum`(coreutils) | `shasum -a 256`(BSD 默认) |
| 文件大小 | `stat -c%s` | `stat -f%z` |

### 10.3 共享代码

- **`db/apply.sh`** · 加 sha256 portability shim:`sha256_of()` 优先用 `sha256sum`,fallback `shasum -a 256`,两者输出格式相同(`<hash>  <file>`)
- **`deploy/maven-settings.xml`** · 阿里云 mirror · macOS 也直接受益
- **数据库迁移逻辑** · `schema_history` 表 + checksum 校验跨平台一致
- **`.checksum-overrides`** · 跨平台共享

### 10.4 不做(macOS)

- 自动备份定时器(用户自己 `crontab -e` 加 `mysqldump | gzip > ~/finance/backup/...sql.gz`)
- HTTPS / certbot(macOS 不暴露公网)
- sudoers / 系统用户隔离(单用户本机环境)
- nginx 反代 / :80 占用
- `/health` 启动后无 systemd 自动重启 · 用户挂前台或 launchd
