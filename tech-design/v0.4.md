## 0. 架构选型一览

| 项 | 选择 | 备选 | 选定理由 |
|---|---|---|---|
| 通胀对照实现 | 前端 Chart.js dataset 加一条 deflate 后线 | 后端预算实际购买力时序 | 前端 deflate 公式简单(`real = nominal / (1+cpi)^t`)· 无新 endpoint · 后端只暴露 cpi_assumption |
| CPI 数据源 | 用户假设值(family.cpi_assumption · 默认 2.0%)| 接国家统计局 API / 写死 | 实时 API 不稳定 · 国情用户认知最关键 · 假设值可控可解释 |
| 基准对照口径 | 直接用 v0.2 `product_category.benchmark_pct` 字段 | 新表存历史 benchmark · 时变率 | benchmark_pct 已是 v0.2 弹药 · 长期年化用户认知合一 · 时变率过度工程 |
| 家庭基准计算 | 按账户余额加权平均 | 按账户类型预设 / 按目标加权 | 加权平均最直观 · 跟用户对"我家配置"心智一致 |

| 还款方式 | v0.4 只做等额本息 | 同时支持等额本金 | 主流场景 · 等额本金占少数 · v0.5 加 |
| /reports 流水图删除策略 | 直接删模板代码 | feature flag 渐进下线 | v0.3 已 backward compat 严守 · v0.4 是用户主动确认要砍 · 直接干净 |

## 2. 关键决策详解

### 决策 33 · 通胀对照实现方式

**问题**:把"实际购买力"线加到 dashboard 净资产趋势图。有两种实现方向。

**备选**:
- A. **前端 Chart.js dataset 加一条 deflate 后的线**(选)
- B. 后端预计算 deflated 时序,存进 `period_snapshot.real_balance` 新列

**取舍**:
- A:deflate 公式 `real = nominal / (1+cpi)^t` 纯前端 5 行 JS · 0 schema · 用户改 CPI 假设秒响应 → 选
- B:每次 cpi_assumption 变要重算整张表 · 不适合"用户可调"

**实施**:
- 后端 `DashboardController` 注入 `family.cpiAssumption` 到 model
- 前端 `cpi-line.js` 读 nominalSeries + cpi · 算 realSeries · push 到 Chart.js datasets
- 用户在 dashboard 顶部点 `2% / 3% / 5%` 触发 `PATCH /admin/family/cpi` AJAX · 重渲染图(不刷页)

### 决策 34 · CPI 数据源

**问题**:CPI 用什么值?

**备选**:
- A. **用户假设值**(family.cpi_assumption · 默认 2.0%)(选)
- B. 接国家统计局 API · 实时拉
- C. 写死单一值(2%)不可改

**取舍**:
- A:用户认知主权 · 退休规划保守可填 3% · 激进可填 1.5% · 可解释性强 → 选
- B:API 不稳定 + 跨境 / 多年份口径混乱 + 实际购买力是"未来假设"不是历史值
- C:刚性 · 用户无掌控感 · 不同家庭通胀感知差异极大(一线 vs 三线)

### 决策 35 · 基准对照口径

**问题**:跑赢/输基准的"基准"用什么值?

**备选**:
- A. **直接用 v0.2 `product_category.benchmark_pct`**(选)
- B. 新建 `benchmark_history` 表 · 存历史时变率
- C. 用户每个账户单独录入 benchmark

**取舍**:
- A:v0.2 早就有(沪深 300 / 标普 500 / 中债等长期年化)· 现成弹药 · admin 可微调 → 选
- B:过度工程 · 中产长期年化关注的是"5-10 年口径"不是月度
- C:录入负担 · 跟"10 分钟/月"硬约束冲突

### 决策 36 · 家庭基准计算口径

**问题**:家庭整体跑赢/输,基准怎么算?

**备选**:
- A. **按账户余额加权平均**(选 · `Σ(account.endBalance × pc.benchmark_pct) / Σ(account.endBalance)`)
- B. 按账户**类型**预设(STOCK→6%, WEALTH→3%, CASH→0.5%)
- C. 按家庭目标加权(退休 50% · 教育 30% · 应急 20%)

**取舍**:
- A:跟"我家钱分在哪 × 各自的基准"心智一致 · 任何账户变动自动反映 → 选
- B:粗暴 · 浪费 v0.3.3 product_category 精细化
- C:目标 weights 是主观的 · 跟"家庭整体基准"概念不匹配

**fallback**:account 没 product_category_code 的,用 B(类型预设)兜底

### 决策 37 · 配置 diff 模板存储

**问题**:4 个模板(标普 4321 / 雪球三分稳健 / 激进 / 永久组合)放哪?

**备选**:
- A. **静态预置表 `allocation_anchor` + 4 行 seed**(选)
- B. 写死代码 enum
- C. JSON 配置文件

**取舍**:
- A:跟 v0.2 `product_category` 风格一致 · admin 可微调 · 用户自定义放 `family.allocation_anchor_custom` JSON 列 → 选
- B:代码改动 · 不灵活
- C:配置文件需要重启 · 不友好

### 决策 38 · AI 调仓 prompt 触发时机

**问题**:AI 调仓建议什么时候生成?

**备选**:
- A. **用户主动按钮点击 · 节流 30 天**(选)
- B. 每月关周期后自动生成
- C. 每次访问 reports 都实时生成

**取舍**:
- A:跟"主动决策"场景匹配 · LLM 成本可控 · 用户感知更强 → 选
- B:不需要被 nag(月报已经是 v0.3b)· 调仓不是常态行为
- C:LLM 成本 ↑ · 每次访问花 1-2s 等待 · 数据没大变化时无意义

**节流实现**:`rebalance_advice_cache` 表 (family_id, anchor_code) UNIQUE · 30 天 TTL · 30 天内同 anchor 直接返缓存

### 决策 39 · AI 调仓 LLM 输出格式

**问题**:LLM 怎么把"调仓建议"传回 UI?

**备选**:
- A. **JSON `{actions:[{from,to,amount,reason}], narrative}` · OutputValidator.checkAndParseJson**(选)
- B. 纯文本 · markdown 渲染
- C. 文本 + 后端 regex 抓 actions

**取舍**:
- A:结构化 actions 可逐条可视化 · v0.3 OutputValidator 已支持 JSON schema 校验 + 真名脱敏 + 担保性词拒绝 → 选
- B:UI 渲染麻烦 · 难自动化检查/解析
- C:正则脆 · LLM 输出格式漂移会断

**Prompt 长度估算**:1200-1500 tokens(家庭 10 账户 + 配置 + 目标 + 风险偏好)· 输出 800-1000 tokens

### 决策 40 · 应急金不闲置算法

**问题**:"应急金充足 + 闲置"怎么判定?

**备选**:
- A. **LIQUID 余额 > 应急需求 × 1.5x 触发**(选)
- B. LLM 实时判断
- C. 用户手动设阈值

**取舍**:
- A:纯计算 · 0 LLM 成本 · 1.5x 阈值跟 v0.2 体检规则一致风格(留 buffer 而不是卡死)→ 选
- B:LLM 成本 · 这种判定没必要
- C:多一个录入字段 · 与 10 min/月 冲突

**阈值由来**:
- `应急需求 = 月均支出 × 6`(v0.3 FR-50c 应急目标默认)
- `1.5x` = 给用户留 50% buffer(避免月支出小幅波动就触发)
- 超出 50% 才提示 = "你的钱在活期睡觉太久"

**触发频率**:dashboard / checkup 实时检查 · 不需要持久化

```sql
-- =========================================================
-- V21 · v0.4 家庭决策辅助字段
-- FR-61a · cpi_assumption 通胀对照线假设值
-- FR-62a · allocation_anchor + custom 配置锚 + 自定义

ALTER TABLE family
  ADD COLUMN cpi_assumption DECIMAL(5,2) NOT NULL DEFAULT 2.00
    COMMENT 'FR-61a · 通胀对照线假设值(% · 默认 2.00 = 2%)· 用户可改 1.5/2/3/5'
    AFTER period_type,
  ADD COLUMN allocation_anchor VARCHAR(32) NOT NULL DEFAULT 'SP_4321'
    COMMENT 'FR-62a · 配置锚 code · 关联 allocation_anchor.code'
    AFTER cpi_assumption,
  ADD COLUMN allocation_anchor_custom JSON NULL
    COMMENT 'FR-62a · 自定义锚 JSON {"cash":10,"invest":30,"property":40,"insurance":20}'
    AFTER allocation_anchor,
  ADD COLUMN risk_appetite VARCHAR(16) NOT NULL DEFAULT 'MODERATE'
    COMMENT 'FR-62b · 家庭风险偏好 · CONSERVATIVE / MODERATE / AGGRESSIVE · LLM 调仓 prompt 输入'
    AFTER allocation_anchor_custom;
```

### 3.3 V23 · rebalance_advice_cache
```sql
-- =========================================================
-- V23 · FR-62b AI 调仓建议缓存(30 天节流)
-- =========================================================
CREATE TABLE rebalance_advice_cache (
  id BIGINT NOT NULL AUTO_INCREMENT,
  family_id BIGINT NOT NULL,
  anchor_code VARCHAR(32) NOT NULL
                          COMMENT '生成时用的锚 · 切换锚会触发新生成',
  content_json JSON NOT NULL
                          COMMENT 'AI 返回 {actions:[...], narrative}',
  prompt_hash CHAR(64) NULL
                          COMMENT 'sha256(prompt) · 数据未变可命中缓存',
  generated_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
  PRIMARY KEY (id),
  UNIQUE KEY uq_family_anchor (family_id, anchor_code),
```

## 6. 风险与缓解

### 6.1 LLM 调仓输出质量
**风险**:LLM 给出"调 ¥100 万到 X"这种不合理金额(超过账户实际余额)
**缓解**:
- Prompt 明确"每个 action 金额必须 ≤ from_account.endBalance × 0.5"
- OutputValidator 加 sanity 校验(amount > 0, from_account 存在, amount <= balance)
- 校验失败 → 删 actions[] 该条 · narrative 保留 · 显警告

### 6.2 4 类目映射不全
**风险**:某账户 product_category=GOLD,在"现金/投资/房产/保险"4 类目里映射到"投资"是否合理?
**缓解**:
- product_category → 4 类目映射表写死 + 文档明确
- GOLD / CRYPTO / FUTURES 全归"投资"(用户认知一致)
- 未设类目的账户按 AccountType 兜底(STOCK→投资,CASH→现金,PROPERTY→房产,WEALTH→投资,LOAN→负 invest)
- v0.5 引入保险类目时,allocation 计算自动反映

### 6.3 砍图引起用户反弹
**风险**:有用户喜欢看瀑布图或桑基图,砍了不开心
**缓解**:
- v0.3.3 prod 用户基数小(用户本人 + 妻子)· 风险低
- 在 reports 顶部加一行说明"v0.4 重塑产品定位 · 把流水图砍了 · 想看流水去 /entry · 这里专注摸清+决策"
- 文档明确"砍掉的图见 v0.3 preview 留作历史"

### 6.4 配置 diff 计算性能
**风险**:每次访问 reports 都计算配置 diff(扫所有账户余额),性能压力
**缓解**:
- 数据已在 v0.1 fact_view 里 · O(账户数) 内存计算 · 单家庭 ≤ 15 账户 · 微秒级
- 无需缓存

### 6.5 自定义 anchor JSON 校验
**风险**:用户填 cash:50, invest:50, property:50(>100%)
**缓解**:
- 前端 JS 实时校验 4 类目和 = 100% · 否则禁用保存按钮
- 后端 service 二次校验 · 越界抛 IllegalArgumentException

### 6.6 提前还贷决策器边界
**风险**:用户填 r_loan=0, r_invest=0 → 公式除零
**缓解**:
- RefinanceNpvCalculator 入参校验 · r ∈ (0, 0.5) · 否则抛 IllegalArgumentException
- UI 表单 min/max 限制 + 默认值兜底
