### 0.2 渐进增强原则

- **检测优先**:UA 检测、`navigator.standalone`、`localStorage` 各自独立,任何一个失败时整个引导静默不出现
- **无 JS 退化**:JS 加载失败时,页面与 v0.1 完全等同,只是没有引导浮层 — 用户仍然可用
- **不阻塞首屏**:引导脚本 `defer` 加载,DOM 就绪后再决定是否注入浮层
- **可关闭**:用户 dismiss 后写 localStorage,有 TTL(微信 24h / iOS 7d),过期后再次提醒一次,**永远可以再 dismiss**
- **DOM 安全**:全部用 `createElement` / `createElementNS` + `setAttribute` / `textContent` 构造节点;**不用 `innerHTML`**(防 XSS + 工具链一致)

## 1. FR-33 · 微信内浏览器引导

### 1.1 触发逻辑

```javascript
function shouldShowWeChatGuide() {
  if (!/MicroMessenger/i.test(navigator.userAgent)) return false;
  const dismissedAt = +(localStorage.getItem('wx_dismissed_at') || 0);
  const TTL_MS = 24 * 60 * 60 * 1000; // 24h
  return Date.now() - dismissedAt > TTL_MS;
}
```

**关键决策**:

- **不做白名单页面**:微信内打开任何页面都触发(包括 `/login`)。理由:登录前用户也应该被引导走,否则填密码后还会遇到 cookie 不持久问题。
- **不区分 iOS 微信 / Android 微信**:UA 都含 `MicroMessenger`,行为相同。
- **不区分 X5 / WKWebView**:无意义 — 微信内的限制对两个内核都成立。
- **dismiss = 写时间戳,非布尔**:这样 24h 后能自动恢复提示,无需额外定时任务。

### 1.2 DOM 结构(运行时注入,不预渲染)

引导浮层不预渲染在 HTML 里(避免对其它环境产生 DOM 噪音),完全由 JS 在运行时通过 `document.createElement` 注入。

注入后的 DOM 结构(节点级):

```
<div id="wx-guide-mask" role="dialog" aria-modal="true">       ← 全屏遮罩
  <svg viewBox="0 0 60 76" class="wx-guide-arrow">              ← 指向右上 ⋯ 的金色箭头
    <path d="..." />                                            ← 用 createElementNS 构造
    <path d="..." />
  </svg>
  <div class="wx-guide-card">                                   ← 引导卡
    <h3>📦 在浏览器中打开</h3>
    <p>微信内浏览器对账房功能支持不全...</p>
    <ol>
      <li><span>1</span> 点击右上角的 ⋯ 按钮</li>
      <li><span>2</span> 选择「在浏览器中打开」</li>
    </ol>
    <button>我知道了 · 继续在微信用</button>
    <div>24 小时内不再提示</div>
  </div>
</div>
```

样式使用内联(`element.style.cssText = ...`),避开污染全局 CSS、避开 Tailwind 动态构建的不确定性。动画 keyframes 一次性 `<style>` 注入。

### 1.3 行为细节

| 事件 | 行为 |
|---|---|
| 点击 dismiss 按钮 | 写 `localStorage.setItem('wx_dismissed_at', Date.now())`,移除 mask DOM |
| 点击遮罩本身(非引导卡) | 不响应(避免误触关闭) |
| 按 ESC | 不响应(微信无键盘场景,且家庭用户不依赖) |
| 横竖屏切换 | CSS 用 inset/百分比,自适应 |
| 页面切换 | localStorage 跨会话生效 |

## 2. FR-34 · iOS Safari PWA 添加到主屏

### 2.1 触发逻辑

```javascript
function shouldShowIosPwaBanner() {
  const ua = navigator.userAgent;
  const isIOS = /iPad|iPhone|iPod/.test(ua);
  const isSafari = /^((?!chrome|android|crios|fxios).)*safari/i.test(ua);
  const isWeChat = /MicroMessenger/i.test(ua);
  const inStandalone = window.navigator.standalone === true;
  const dismissedAt = +(localStorage.getItem('pwa_dismissed_at') || 0);
  const TTL_MS = 7 * 24 * 60 * 60 * 1000; // 7d

  return isIOS
      && isSafari
      && !isWeChat
      && !inStandalone
      && (Date.now() - dismissedAt > TTL_MS);
}
```

**关键决策**:

- **必须排除微信内 iOS Safari**:微信内 UA 在 iOS 上同时匹配 Safari 和 MicroMessenger。微信检测优先 — 实际代码里两条分支互斥(微信里只弹微信引导)。
- **`navigator.standalone === true`**:用户已经把账房加到主屏并从主屏点开 — 在 standalone 模式下不再骚扰。
- **延迟 1.5 秒弹出**:`setTimeout(showBanner, 1500)`。理由:首屏加载时间不打扰用户。
- **dismiss TTL 7 天**:比微信引导 24h 长很多。理由:iOS PWA 引导是"建议",不是必要;7 天内被提示一次足够。

### 2.2 DOM 结构(底部固定卡片)

```

  <button class="pwa-close" aria-label="关闭">✕</button>

    <div class="pwa-icon">账</div>      ← CSS 渲染棕青配色,与 brand 一致
    <div class="pwa-text">
      <h3>添加到主屏 · 像 App 一样使用</h3>
      <p>FAMILY · COUNTING · HOUSE</p>
    </div>
  </div>
  <ol>
    <li><span>1</span>点击底部分享按钮 ⤴</li>
    <li><span>2</span>滚动找到「添加到主屏幕」</li>
    <li><span>3</span>点击右上角「添加」</li>
  </ol>
  <div class="pwa-actions">
```

## 3. 共享基建:`mobile-guide.js`

完整脚本结构(~150 行),放在 `src/main/resources/static/js/mobile-guide.js`。

### 3.1 模块结构

```
(IIFE)
  ├── 'use strict'
  ├── Storage 工具(localStorage 不可用降级 no-op)
  ├── UA 检测变量(isWX / isIOS / isSafari / inStandalone)
  ├── injectStyle(id, css)        — 一次性注入 keyframes
  ├── el(tag, css, text?)          — 创建 div/span/button 等普通节点的 helper
  ├── svgEl(tag, attrs)            — 创建 SVG 节点(createElementNS)的 helper
  ├── showWxGuide()                — FR-33 微信引导
  ├── showIosPwaBanner()           — FR-34 iOS PWA banner
  └── DOMContentLoaded 分发
```

### 3.2 关键设计点

| 点 | 决策 | 理由 |
|---|---|---|
| IIFE 包装 | `(function(){…})()` | 避免污染全局 |
| `'use strict'` | 用 | 防变量泄漏到 window |
| localStorage 降级 | try-catch 包装 | iOS 无痕模式 / 隐私沙箱可能抛 |
| **`createElement` 而非 `innerHTML`** | 全部 createElement | 防注入,工具链规则一致(claude-code 安全 hook 拦截 innerHTML) |
| SVG path 用 `createElementNS` + `setAttribute` | 是 | 同上,namespace 必须正确才能渲染 |
| 动画用 inline `<style>` 注入 | 是 | 避开依赖 Tailwind/外部 CSS;动画一次注入复用 |
| 微信 / iOS 互斥 | 微信优先 | 微信内不弹 iOS PWA banner(没意义,微信内不能加主屏) |
| Z-index | 微信 99999 / PWA banner 99998 | 高于一切应用层级,避免被遮挡 |

#### 4.1.2 PNG 生成(三个尺寸)

服务器需安装 `librsvg2-bin`(轻量 ~2MB,Ubuntu/Debian):

```bash
sudo apt install -y librsvg2-bin
```

跑一次性脚本生成 3 张 PNG。新建 `deploy/gen-icons.sh`:

```bash
#!/usr/bin/env bash
# 一次性生成 PWA / iOS 主屏图标 PNG(改了 SVG 后重新跑)
# 用法: bash deploy/gen-icons.sh
set -e

ROOT="$(cd "$(dirname "${BASH_SOURCE[0]}")/.." && pwd)"
SRC="$ROOT/src/main/resources/static/img/apple-touch-icon.svg"
DIR="$ROOT/src/main/resources/static/img"

if ! command -v rsvg-convert &> /dev/null; then
  echo "✗ rsvg-convert not found. Install:  sudo apt install -y librsvg2-bin"
  exit 1
fi

if [[ ! -f "$SRC" ]]; then
  echo "✗ Source not found: $SRC"
  exit 1
fi

rsvg-convert -w 180 -h 180 "$SRC" -o "$DIR/apple-touch-icon-180.png"
rsvg-convert -w 192 -h 192 "$SRC" -o "$DIR/icon-192.png"
rsvg-convert -w 512 -h 512 "$SRC" -o "$DIR/icon-512.png"

echo "✓ 3 PNGs generated. Now: git add + commit + redeploy."
ls -la "$DIR"/{apple-touch-icon-180,icon-192,icon-512}.png
```

**生成的 PNG 提交到 git**(避免每次 deploy 都重跑;只有改 SVG 时才重跑):

```
src/main/resources/static/img/apple-touch-icon.svg       ← 设计源(改这里)
src/main/resources/static/img/apple-touch-icon-180.png   ← iOS 主屏专用(<link rel="apple-touch-icon">)
src/main/resources/static/img/icon-192.png               ← manifest 引用
src/main/resources/static/img/icon-512.png               ← manifest 引用 + maskable
```

> **备选方案**(如果服务器无法装 librsvg2-bin):本地 macOS 用 `qlmanage -t -s 180 apple-touch-icon.svg`,或 Figma / Sketch 手工导出,再 scp 上去。

### 4.2 .webmanifest MIME type(可能需要)

Spring Boot 3.x 默认能识别 `.webmanifest` 扩展名映射到 `application/manifest+json`(自 Spring 5.3+ 起内置)。

**先跑一次** `curl -I http://localhost:20000/manifest.webmanifest`:

- 回 `application/manifest+json` → 不用改
- 回 `application/octet-stream` 或别的 → 在 `WebMvcConfig.java` 加:

```java
@Override
public void configureContentNegotiation(ContentNegotiationConfigurer cfg) {
    cfg.mediaType("webmanifest", MediaType.parseMediaType("application/manifest+json"));
}
```

## 5. 渐进部署 / 回退策略

### 5.1 部署顺序(按风险从低到高)

1. **静态资源先行**:把 `manifest.webmanifest`、`apple-touch-icon.svg`、3 个 PNG 加进 `static/`,提交后部署 — 这一步对所有用户透明,不出现任何 UI 变化。curl 验证 6 个文件 200。
2. **SecurityConfig 放行 + WebMvcConfig 改 MIME**(如需):重启后 curl 验证 `/manifest.webmanifest` 200 + 正确 Content-Type。
3. **layout.html head 改动**:加 5 个 meta + 1 个 link manifest + 1 个 link apple-touch-icon-180.png + 1 个 script defer — 上线后:
   - 桌面浏览器 / Chrome / Firefox / Edge:无任何视觉变化
   - iOS Safari:DevTools 显示"Web App Manifest 检测成功",但 UI 无变化(banner 还没注入,`mobile-guide.js` 还没创建)
4. **mobile-guide.js 上线**:这一步开始有 UI 变化 — 单独 commit,方便快速 revert。

### 5.2 紧急关闭(三层)

| 层级 | 操作 | 影响 | 生效 |
|---|---|---|---|
| **L1 紧急** | 把 `mobile-guide.js` 内容替换成单行 `(function(){})()` | 所有用户立即不再看到引导 | 部署 + buildVersion 自动失效缓存 |
| **L2** | layout.html 注释掉 `<script ..mobile-guide.js>` | 同上,更彻底 | 同上 |
| **L3 完全回退** | revert SecurityConfig + WebMvcConfig + layout.html + 删 manifest + 删 PNG | 完全回到 v0.1 状态 | 全量部署 |

## 7. 风险与缓解

| 风险 | 概率 | 影响 | 缓解 |
|---|---|---|---|
| 微信不同版本 UA 字符串变化 | 低 | 引导误判/漏判 | 用宽松正则 `/MicroMessenger/i`,长期稳定;真出现 patch UA |
