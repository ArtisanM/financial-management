## 1. 产品概述

### 1.1 背景

夫妻两人当前家庭资产分散在多个渠道:股票账户(若干)、银行储蓄/活期(若干)、支付宝/微信钱包/理财、可能的房产或外汇等。日常无系统记录,导致:

- **看不到全局**:无法回答"我们家现在共有多少钱"
- **看不到趋势**:更无法回答"我们家这一年是变富了还是变穷了"
- **分不清来源**:工资攒下来的钱和投资赚的钱混在一起,无从评估投资能力
- **无法异步协作**:夫妻时间错开,缺一个共同载体来"各填各的"

市面上的方案有局限:支付宝/雪球只能看自家平台,国内通用记账 App 不区分本金和收益,海外开源工具(YNAB / Firefly III / Beancount)中文支持弱或学习曲线高,Excel 缺少自动化(尤其是 XIRR)。

### 1.2 为什么自建

仅在以下三件事上,自建相对 Excel/飞书有真实增量价值:

1. 自动 XIRR / 年化收益率(Excel 公式难维护)
2. 移动端友好填报(配偶能 5 分钟独立完成)
3. 双用户协作(国内记账 App 普遍弱)

其余(记数字、汇总、画图)Excel 都能做。所以 v0.1 必须把这三件事做到位,其他一切先砍。

### 1.3 目标

#### 北极星指标

> 夫妻**连续 6 个月**完成月度填报,且每次平均填报时间 ≤ 10 分钟。

低于这条线就视为产品失败 — 因为不可持续。

#### 次级目标

- 任意时刻能在 1 分钟内回答:"现在我们家共有多少钱?"
- 任意时刻能在 1 分钟内回答:"过去一年我们的工资攒下了多少、投资赚/亏了多少?"
- 任意账户的年化收益率(XIRR)随时可看
- 数据可一键导出 CSV,不被工具绑架

### 1.4 目标用户

只有 2 个真实用户。这个事实决定了所有产品设计:

#### 用户 A · 丈夫

- 软件工程师,Java 背景,有服务器和域名
- 系统的开发者和维护者,容忍偶尔的小 bug
- 平时通过桌面浏览器使用居多,移动端次之
- 负责账户:他名下的所有股票账户、他的工资卡、他的支付宝
- 痛点容忍度:高 — 出 bug 自己能修

#### 用户 B · 妻子

- 非技术背景,对电子表格不感冒
- 主要在手机上操作(微信生态、支付宝)
- 负责账户:她名下的所有银行卡、她的支付宝/理财、共同账户(若有)
- 痛点容忍度:**低** — 任何"不知道这格填什么"会立即流失

> ⚠️ **设计原则**:任何 UI 决策需通过"妻子能否独立完成"的判断。这条优先于"功能完整性"。

### 1.5 v0.1 边界

#### In Scope(v0.1 一定做)

| # | 能力 | 备注 |

### 1.6 成功的衡量(具体可观测)

| 指标 | 目标值 | 测量方式 |
|---|---|---|
| 月度填报完成率 | ≥ 95%(连续 6 月) | 每月有快照的账户数 / 当月活跃账户数 |
| 妻子独立填报时长 | ≤ 5 分钟 | 妻子主观计时,系统记录首次提交到最后提交的间隔 |
| 妻子主动求助次数 | ≤ 1 次/月 | 主观统计 |
| 系统可用性 | ≥ 99%(月度) | systemd uptime + nginx 5xx 监控 |
| 数据备份成功率 | 100% | 每周备份 cron job 结果检查 |
| 误差校验通过率 | 100% | NetChange = NetWorth(M) − NetWorth(M-1) 恒等式不能违反 |

## 2. 用户流程

### 2.1 月度主流程总览

```
                    ┌──────────────────────────────────┐
                    │  M月25日:系统标记上月数据"待填"  │
                    └──────────────────┬───────────────┘
                                       ▼
              ┌────────────────────────────────────────────┐
              │  M+1月1-3日:每个账户对应的"月末余额"在    │
              │  券商/银行/支付宝 App 上可查到            │
              └────────────────────────────────────────────┘
                                       │
              ┌────────────────────────┴────────────────────────┐
              │                                                  │
              ▼                                                  ▼
   ┌──────────────────┐                              ┌──────────────────┐
   │  丈夫流程(异步) │                              │  妻子流程(异步) │
   │  浏览器/手机均可  │                              │  手机为主         │
   │  填自己的账户    │                              │  填自己的账户     │
   └──────────────────┘                              └──────────────────┘
              │                                                  │
              └────────────────────┬─────────────────────────────┘
                                   ▼
                  ┌──────────────────────────────────────┐
                  │ 系统检测"双方都已完成本月所有账户" │
                  │ Dashboard 解锁本月完整图表          │
                  └──────────────────────────────────────┘
```

**关键约束**:不要求两人同时在线。系统状态对夫妻双方实时同步。

### 2.2 时间锚点

| 时间 | 触发事件 |
|---|---|
| 每月 1 日 凌晨 2:00 | 系统拉取上月末汇率(USD→CNY、HKD→CNY) |
| 每月 1 日 9:00 | 系统标记"M-1 月待填",首次站内提醒 + 邮件 |
| 每月 5 日 9:00 | 第二次提醒(若仍有缺漏) |
| 每月 10 日 9:00 | 第三次提醒 + 标黄"数据迟报" |
| 每月 15 日 | 之后填报标"⚠️ 迟填,XIRR 时间精度受影响" |

### 2.3 丈夫流程(详细)

#### 触发

- 早晨打开邮箱,看到"本月待填报"邮件;或
- 主动想看"上个月赚了多少",访问 finance.example.com

#### 步骤

1. **登录**
   - 桌面:浏览器记住密码,一键登录
   - 移动:浏览器收藏,Spring Security 用 RememberMe Cookie(30 天)

2. **进入填报页** `/entry?month=2026-04`(默认上月)
   - 顶部:月份选择器(可改为更早的月做补录)
   - 顶部:本月填报进度条 "已完成 3 / 7 个账户(妻子已填 2 / 4)"
   - 列表:**默认只展开自己负责的账户**;对方负责的账户折叠成只读小条

3. **逐账户填报**(以"招行储蓄卡-工资"为例)
   - 在另一个 Tab 或手机上打开招行 App,看月末余额(假设 ¥58,200)
   - 回到本系统,点击账户行展开
   - 看到 placeholder:`上月末 ¥45,000 · 较上月 ?`
   - 输入"月末余额":`58200`(系统实时显示"较上月 +¥13,200")
   - 想想:这个月有几笔工资入账?哪些是消费?
     - 输入"本月工资性流入":`15000`(category 选"工资")
     - 输入"本月消费性流出":`1800`(category 选"消费")
   - 点击"提交本行"(或 HTMX 自动提交 onblur)
   - 行首图标变 ✓ 绿色

4. **登记跨账户转账**(以"招行→证券"为例)
   - 系统不强制每行都填工资/消费,但**强制推断转账**:如果某账户余额变化与"工资−消费"出入很大,系统弹提示:"招行储蓄卡变化 −¥10,000,看起来像是转出?是否登记一笔转账?"
   - 点"添加转账",弹层:
     - 转出账户:`招行储蓄卡-工资`(本行预填)
     - 转入账户:`华泰证券`(下拉选)
     - 金额:`10000`
     - 备注:`季度调仓加仓`
   - 提交 → 同时更新两个账户的"内部转账"状态

5. **完成自己的部分**
   - 顶部进度条:"丈夫已完成 4/4 · 妻子待完成 4 个账户"
   - Dashboard 上仍标"本月数据未完整,部分图表为预览态"

#### 决策点:外部现金流可以省略吗?

可以。但有取舍:

- **完整填**(推荐):月度损益分解为"工资 / 消费 / 投资损益"三部分,瀑布图清晰
- **只填余额**:省事,但所有变化都被归到"投资损益",会高估投资能力或低估消费

UI 上对这种情况给软提示:**"未登记任何外部现金流,本月余额变化将全部记为投资损益。确认无收支吗?"** — 可一键确认跳过,但不会默默吞掉。

### 2.4 妻子流程(详细)

#### 设计原则

> 假设她对系统毫无兴趣,只想 5 分钟搞定月底"那件事"。所有 UI 优先服务这个假设。

#### 触发

- 收到邮件,点击邮件中的"现在去填"按钮 → 落地登录页 → 登录后直接到自己的填报视图

#### 步骤

1. **登录**(手机)
   - 同上,RememberMe 30 天免输密码

2. **直接落地"我的本月待填"** `/entry?month=2026-04&mine=true`
   - **过滤掉丈夫的账户**,只显示自己的 4 个账户
   - 顶部一句话提示:"4 张账户,大约 5 分钟"
   - 没有月份选择器(默认本月,简化)

3. **逐账户**(以"妻子的支付宝-余额宝"为例)
   - 大字号卡片视图(不是表格),每张卡片占满屏宽
   - 卡片内容:
     ```
     ┌─────────────────────────────┐
     │ 💰 支付宝-余额宝             │
     │ 上月末 ¥12,300              │
     │                             │
     │ 月末余额                    │
     │ [           ¥                ] │  <- 大输入框,数字键盘
     │                             │
     │ 本月工资/收入(可选)        │
     │ [           ¥                ] │
     │                             │
     │ 本月支出(可选)             │
     │ [           ¥                ] │
     │                             │
     │ [跨账户转账?] [✓ 提交本行]   │
     └─────────────────────────────┘
     ```
   - 输入"月末余额":打开支付宝余额宝页面 → 看到 ¥12,800 → 回输入框填
   - "本月工资":如果工资发到这张卡,填;否则跳过
   - "本月支出":通常她记不清具体数字,**允许跳过**;系统会暗示这会让"投资损益"看起来比真实高
   - 提交 → 卡片变绿色 ✓,自动滚到下一张

4. **退出**
   - 全部卡片绿色后,落到"完成"页:"本月你已完成 4 张账户,谢谢!丈夫还有 1 张未完成。"

#### 妻子流程的设计取舍

- ❌ **不要**展示 XIRR、瀑布图等技术图表 — 她只关心"填完了吗"
- ❌ **不要**让她看到对方账户的细节
- ✅ 默认所有非"月末余额"字段为可选,空值由系统智能处理
- ✅ 一行错误用红色文案而非弹窗(弹窗在小屏幕上反感)
- ✅ 提交后立刻反馈,**不要**让她不确定是否保存了

### 2.5 具体场景:登记本月薪资

#### 假设

- 丈夫月薪 ¥30,000,每月 15 日发到"招行工资卡"
- 妻子月薪 ¥18,000,每月 28 日发到"建行储蓄卡"

#### 操作

**丈夫这边**:
- 在"招行工资卡"行的"本月工资性流入"填 `30000`,category 默认"工资"
- 不需要填具体日期(月度颗粒度)
- 如果丈夫还有"年终奖"或"绩效奖金",可在同一账户**追加一笔** `INCOME` 记录,category 选"奖金"
  - 系统支持每个账户每月任意笔 cash_flow,但**默认 UI 只显示一笔**;点"+"能添加更多
- 月末余额仍按招行 App 实际显示填(不需要自己算"上月末 + 工资 − 消费")

**妻子这边**:同样在自己卡上填"工资性流入" `18000`。

#### 系统怎么用这个数据

- `cash_flow(account_id=招行工资卡, month=2026-04, kind=INCOME, amount=30000, category='工资')`
- 月度损益计算:
  ```
  PnL(招行, 4月) = 期末余额 − 上月末余额 − 工资入账 + 消费支出 − 转账净流入
  ```
- 计入家庭维度的"工资性流入"= 30000 + 18000 = 48000(瀑布图最左一根)

#### 如果忘填工资

后果:**这部分被归到"投资损益"**,瀑布图错。
解决:任意时刻可以打开历史月份回填,系统会重新计算这一月之后的所有 XIRR。

### 2.6 具体场景:更新某个余额

最常见的几种情况:

#### 情况 A · 月底正常更新

如 § 2.3 / § 2.4 所述,在 `/entry` 上填即可。

#### 情况 B · 月中突然想看一下当前总资产

v0.1 不支持月中实时余额。Dashboard 显示的总是"上月末"数字。
解决方法:在月中也可以打开 `/entry?month=2026-05` 提前填一个粗略数字,Dashboard 会显示但标"本月数据未冻结";月底再来重填即可(同 month + 同 account 的 snapshot 会被覆盖,要二次确认)。

#### 情况 C · 发现上月填错了

- 打开 `/entry?month=2026-03`(选历史月)
- 修改月末余额或现金流
- 提交时系统提示:"修改 2026-03 数据将重新计算 2026-03 起所有账户和总资产 XIRR / 损益。确认?"
- 确认后所有图表重新算

#### 情况 D · 新开了一个账户

- 先到 `/accounts` 添加账户(填名称、类型、币种、负责人、显示顺序)
- 然后回 `/entry`,看到新账户出现在列表里,填月末余额即可
- 如果是用现有账户的钱新开的(如:从招行存了一笔到证券),**还要登记一笔跨账户转账**

#### 情况 E · 销户了

- 在 `/accounts` 把账户标"归档"(不删,保留历史数据)
- 销户前的最后一个月,余额填 0,登记一笔"转账"到承接账户
- 之后月份的填报页不再显示这个账户

### 2.7 跨账户转账登记规则

#### 谁登记?

**约定**:由"转出账户的负责人"登记。但系统**不强制**,任何一方都能登记,通过 `transfer` 表的唯一性保证不重复(同一对账户、同金额、同月份、间隔 < 24 小时的会触发"是否重复登记"二次确认)。

#### 跨币种转账?

例如:从"招行美元账户"换汇 ¥1 万到"招行人民币活期"。
- 登记两条 transfer? 不,只登记一条
- 用一条 transfer + 自动计算汇率损益:
  - `transfer(from=美元卡, to=人民币卡, amount=1428.57 USD)` // 假设当时汇率 7.0
  - 系统自动按月末汇率折算,差异计入"汇率损益"(显示在月度瀑布图)
- v0.1 简化:暂不区分"汇率损益",直接记一条转账,差异自动归入"投资损益"

#### 房贷月供等于"转账+消费"?

- 不是。房贷月供属于 EXPENSE(对家庭来说是钱离开了系统,变成"住房")
- 如果你已经把"房产"作为一个 PROPERTY 类账户,那就是:
  - 招行卡:EXPENSE 5000(还月供)+ 余额减少
  - 房产账户:不变(月度估值不变,除非要严肃追踪房价)

### 2.8 提醒与缺漏处理

#### 站内提醒

- 登录后 Dashboard 顶部红色 banner:"还有 2 个账户本月未填:招行储蓄卡、华泰证券"
- 点击 banner 直接跳到对应填报行

#### 邮件提醒

- 月度第 1、5、10 天发邮件
- 邮件里直接列出待填账户 + 一键登录链接

#### 缺漏的影响

- 总资产 NetWorth(M):该月只汇总有 snapshot 的账户,UI 标"⚠️ 数据不完整"
- 月度损益 PnL_total(M):同上
- 该账户的 XIRR:只用已有数据点算,数据点 < 3 个则显示"—"

### 2.9 异常场景

| 场景 | 处理 |
|---|---|
| 妻子忘了密码 | 丈夫后台 SQL 重置 bcrypt hash(v0.1 不做"找回密码"页) |
| 服务器挂了 | systemd 自动重启;每周备份保证数据可恢复 |
| 汇率 API 挂了 | 当月汇率字段空,UI 标"⚠️ 本月汇率缺失",提供手填覆盖 |
| 同账户同月填报两次 | 后填覆盖前填,但有"上次填的是 ¥X,XXX,确认覆盖?"二次确认 |
| 妻子误删账户 | v0.1 不允许"删除"账户,只能"归档";归档可一键恢复 |

## 4. 指标计算口径

本节严格定义系统所有指标的公式、数据来源、边界处理和校验恒等式。所有指标都从 4 张原始表计算而来,**不落库**(除汇率)。

### 4.1 符号约定

| 符号 | 含义 |
|---|---|
| `M` | 月份(用月份首日 `YYYY-MM-01` 表示) |
| `M-1` | 上一个月 |
| `A` | 账户(account) |
| `C` | 币种(CNY / USD / HKD) |
| `currency(A)` | 账户 A 的币种 |
| `type(A)` | 账户 A 的类型(STOCK/CASH/WEALTH/PROPERTY/OTHER) |
| `B(A, M)` | 账户 A 在月份 M 的月末余额(原币种,从 monthly_snapshot.end_balance) |
| `I(A, M)` | 账户 A 在月份 M 的 INCOME 类 cash_flow 之和(原币种) |
| `E(A, M)` | 账户 A 在月份 M 的 EXPENSE 类 cash_flow 之和(原币种) |
| `T_in(A, M)` | 月份 M 内,转账目标 = A 的金额之和 |
| `T_out(A, M)` | 月份 M 内,转账源头 = A 的金额之和 |
| `FX(C, M)` | 月份 M 末,1 单位 C 折成 CNY 的汇率(从 fx_rate;CNY→CNY = 1) |
| `Active(M)` | 月份 M 时未归档(`archived_at IS NULL OR archived_at > M末日`)的账户集合 |

### 4.2 指标 1 · 账户月度投资损益(原币种)

```
PnL(A, M) = B(A, M) − B(A, M-1) − [I(A, M) − E(A, M)] − [T_in(A, M) − T_out(A, M)]
```

**直觉**:这个月余额变了多少,把外部流入流出和内部转账剔掉,剩下就是市场涨跌带来的损益。

**边界**:
- `B(A, M-1)` 不存在(账户首月或缺数据):`PnL = null`,UI 显示"—"
- 跨币种转账:`T_in / T_out` 仍按本币种金额参与;v0.1 不单独折算汇率损益

**校验**:
- 对纯现金账户(如银行储蓄),`PnL` 应接近 0(只有少量利息)
- 测试用例:1 月余额 ¥10,000,2 月余额 ¥10,500,工资入 ¥0,消费 ¥0,转账 ¥0 → PnL = ¥500(可能是利息或对账误差)

### 4.3 指标 2 · 账户月度投资损益(折 CNY)

```
PnL_CNY(A, M) = PnL(A, M) × FX(currency(A), M)
```

**说明**:用月末汇率折算。理论上更精确的做法是用 TWR 或月内多点平均,但月度颗粒度下不必要。

### 4.4 指标 3 · 月度家庭总损益

```
PnL_total(M) = Σ PnL_CNY(A, M)  for A ∈ Active(M)
```

跳过 `PnL(A, M) = null` 的账户,但 UI 标"⚠️ 数据不完整"。

### 4.5 指标 4 · 月度家庭总资产

```
NetWorth(M) = Σ [ B(A, M) × FX(currency(A), M) ]  for A ∈ Active(M)
```

**边界**:
- 某账户 `B(A, M)` 缺失:跳过该账户,UI 标"⚠️"
- 某币种 `FX(C, M)` 缺失:整月 NetWorth 标红"⚠️ 汇率缺失",提示去 `/admin/fx` 手填

### 4.6 指标 5 · 资产配置

```
Allocation(M, t) = Σ [ B(A, M) × FX(currency(A), M) ]  for A ∈ Active(M) where type(A) = t
比例(M, t) = Allocation(M, t) / NetWorth(M)
```

环形图按 `t` 分扇区显示比例。

### 4.7 指标 6 · 账户级 XIRR

XIRR = 不规则现金流的内部收益率,年化。

#### 现金流序列构造

对账户 A 在时间窗口 `[M_first, M_last]` 内:

```
现金流序列 CF = []

// 期初:首月初始投入(视为流出账户的资金来源,符号为负)
CF.append( (date=M_first 月末日, amount = -B(A, M_first)) )

// 中间各月:净外部流入(流入账户为负,流出为正)
for M in (M_first, M_last]:
    netExternal = I(A, M) − E(A, M) + T_in(A, M) − T_out(A, M)
    if netExternal != 0:
        CF.append( (date=M 月末日, amount = -netExternal) )

// 期末:当前账户价值(视为回收,正数)
CF.append( (date=M_last 月末日, amount = +B(A, M_last)) )
```

注意:期初和期末刚好抵消的现金流要保留两条不能合并(数学上不影响,但代码清晰)。

#### XIRR 求解

求 `r` 使得:

```
NPV(r) = Σ CF_i.amount / (1 + r)^((CF_i.date − CF_0.date) / 365.0) = 0
```

**实现**:
- Apache Commons Math `org.apache.commons.math3.analysis.solvers.BrentSolver`
- 区间:`[-0.99, 10.0]`(覆盖 -99% 到 +1000% 年化)
- 最大迭代:100
- 容差:1e-7

#### 边界

| 情况 | 处理 |
|---|---|
| 数据点 < 12 个月 | 不算年化,改算"累计收益率" = `(终值 / 初值) − 1`,UI 标"未满 12 月" |
| Brent 求根失败(无解或多解) | 返回 null,UI 显示"—",tooltip:"现金流模式无法计算 XIRR" |
| `B(A, M_first) = 0` 且首月有大额流入 | 把首笔大额流入作为初始投入(改用 first-flow-as-baseline 策略) |
| 全期数据缺失 > 50% 月份 | 返回 null,UI 显示"—"+ ⚠️ |

#### 测试用例(必须通过)

| Case | 输入 | 期望 XIRR(±0.5%) |
|---|---|---|
| 1 | 1 月初投 ¥10,000,1 年后值 ¥11,000,无中间流 | ≈ 10% |
| 2 | 1 月初投 ¥10,000,7 月加仓 ¥5,000,1 年后值 ¥17,000(累计 +¥2000) | ≈ 16% |
| 3 | 纯持有 ¥10,000 → ¥10,000,1 年无变化 | = 0% |
| 4 | 1 月投 ¥10,000,3 月赎回 ¥10,000,3 月再投 ¥10,000,1 年后值 ¥10,000 | ≈ 0% |

### 4.8 指标 7 · 总资产级 XIRR

把所有账户的外部现金流和总资产折成 CNY 后,套用 § 4.7 的方法:

```
家庭维度现金流序列:
  CF_0 = (M_first 月末日, amount = -NetWorth(M_first))
  for each month M ∈ (M_first, M_last]:
      familyExternal_CNY(M) = Σ [ (I(A,M) − E(A,M)) × FX(currency(A), M) ]
                              for A ∈ Active(M)
      // 注意:跨账户转账在家庭维度互相抵消,不出现在序列里
      if familyExternal_CNY(M) != 0:
          CF.append( (M 月末日, -familyExternal_CNY(M)) )
  CF_last = (M_last 月末日, +NetWorth(M_last))
```

**关键**:跨账户 `transfer` 在家庭维度互相抵消,所以 `T_in / T_out` 不出现在家庭现金流里。这是账户级 transfer 表设计的核心价值。

### 4.9 指标 8 · 月度收支瀑布

```
ExternalIncome(M)   = Σ [ I(A, M) × FX(currency(A), M) ]  for A ∈ Active(M)
ExternalExpense(M)  = Σ [ E(A, M) × FX(currency(A), M) ]  for A ∈ Active(M)
InvestmentPnL(M)    = PnL_total(M)
NetChange(M)        = ExternalIncome(M) − ExternalExpense(M) + InvestmentPnL(M)
```

瀑布图四根柱子:基线(上月 NetWorth)→ +工资 → −消费 → ±投资损益 → 终点(本月 NetWorth)。

### 4.10 校验恒等式(每次计算后必须成立)

```
NetWorth(M) − NetWorth(M-1)
  = ExternalIncome(M) − ExternalExpense(M) + InvestmentPnL(M)
  = NetChange(M)
```

**实现**:`PnlCalculator.assertIdentity(M)` 在每次月度计算后断言,误差 > ¥0.01 抛 `DataInconsistencyException`,UI 标红警示。

**常见违反原因**:
1. 跨账户转账漏登记(资金"凭空消失"或"凭空出现") → 提示用户检查 transfer
2. 某账户 snapshot 缺失 → 提示去填
3. 跨币种汇率折算精度问题(罕见,< ¥1) → 容忍

### 4.11 数据来源与计算时机

| 数据 | 来源 | 计算/更新时机 |
|---|---|---|
| 月末余额 `B(A, M)` | 用户填(`monthly_snapshot`) | 用户提交时 |
| 现金流 `I, E` | 用户填(`cash_flow`) | 用户提交时 |
| 转账 `T_in, T_out` | 用户填(`transfer`) | 用户提交时 |
| 汇率 `FX` | exchangerate.host(`fx_rate`) | 每月 1 日凌晨 cron + 手填覆盖 |
| `PnL, XIRR, NetWorth, Allocation` | 上述 4 张表实时计算 | Dashboard / Reports 请求时 |

**性能**:数据量小(2 人 × ~10 账户 × 12 月/年 = 一年 ~120 行 snapshot),全部在内存里 SQL+Java 计算,毋须缓存。

### 4.12 精度与数值类型

| 用途 | 类型 | 小数位 |
|---|---|---|
| 余额 / 现金流 / 转账 | `DECIMAL(18, 2)` | 2(分) |
| 汇率 | `DECIMAL(18, 6)` | 6 |
| 折算中间值 | `BigDecimal`,scale=6 | 6 |
| 显示值 | 四舍五入到 2 位 | — |

**禁止**:用 `double` / `float` 做金额计算(浮点误差会破坏 § 4.10 恒等式)。

| 短信 / 微信公众号通知 | v0.1 仅站内 + 邮件即可;微信公众号需要个人订阅号注册等成本 | v0.2 视情况 |
| 双因素认证 / OAuth | 系统只有 2 个内部用户,密码 + 服务器防火墙已够 | 未来不做 |
| 数据加密(磁盘以外) | MySQL 已存在受信服务器内,异地备份可用 GPG 加密 | v0.2 增强备份加密 |
| 移动端 PWA / 离线填报 | v0.1 假设填报时有网络 | v0.3+ |
| 复杂权限分级 | 2 人之间无权限差异 | 未来不做 |
| 资产负债表 / 损益表(会计风) | 受众非财务专业,过度复杂 | 未来不做 |
| 投资标的对标(沪深 300、CSI 等) | 账户级颗粒度无法做精确对标 | 未来不做(需持仓级) |
| 房贷/车贷的还款计划展示 | 月度 EXPENSE 已经够用;不做摊销表 | v0.3+ 可加"长期负债"模块 |

### 5.2 已知的"v0.1 可接受的妥协"

这些是**有意识地接受的限制**,使用中要清楚:

1. **跨币种汇率损益不单独标识**:转入转出的汇率差异自动归入"投资损益",会让美元账户的"投资损益"包含汇率波动部分。**理由**:v0.1 简化;v0.2 可加"汇率损益"独立列。

2. **月内多笔现金流不区分时间**:同月几笔工资 / 几笔消费合并成一笔。**理由**:满足"10 分钟/月"约束;XIRR 时间精度月度颗粒度足够。

3. **TWR 不单独计算**:只算 XIRR(=DWR=资金加权)。**理由**:对家庭场景 IRR 更直观("我赚了多少")而非"投资标的本身的表现"。

4. **历史数据不可机器导入**:从本月起点。**理由**:v0.1 简化;后期加 CSV 导入工具(模板由系统提供)。

5. **Dashboard 不实时(月颗粒度)**:总资产是"上月末"快照,月中无法看到真实当下。**理由**:契合"月度记账"产品定位;实时需要券商 API 直连(已不做)。

6. **没有"撤销最后一次提交"按钮**:覆盖型修改 + 二次确认替代。**理由**:简单;有问题可手填修正。

### 5.3 路线图(粗略)

#### v0.2(预计 v0.1 稳定运行 6 月后回看)

- **汇率损益独立标识**(从"投资损益"中拆出)
- **CSV 历史数据导入**(让用户能补录过往年份)
- **数据备份 GPG 加密**(异地存储更安全)
- **月度对比报告自动邮件**(每月 5 号生成上月报告 PDF / HTML 邮件)
- **微信公众号推送**(如果邮件不够用)
- **Dashboard 自定义**(用户可拖拽 KPI 卡片)

#### v0.3(更远,视真实需求)

- **持仓级颗粒度**(可选模块)— 数据模型已留扩展位 `holding` / `holding_snapshot`
- **房贷/车贷长期负债跟踪** — 摊销表、还款进度
- **孩子账户 / 父母代管账户** — 多用户支持

#### 永远不做

- 复杂的预算包络系统
- 个股级买卖记录(违反"10 分钟"原则)
- 给陌生用户使用的多租户(永远只是夫妻 2 人的工具)

### 5.4 评估"是否要进 v0.x"的 checklist

新功能进入下个版本前,必须通过:

- [ ] 不增加月度填报时间(或增量 < 1 分钟)
- [ ] 不让妻子看不懂(她能在 30 秒内理解功能用途)
- [ ] 用了真实数据 ≥ 6 月后才提需求(避免拍脑袋功能)
- [ ] 对应 Excel/飞书有"明显不可能或太麻烦"的对比理由
- [ ] 实现工作量 ≤ 8 小时(否则拆小)

### 5.5 失败信号(触发就要回炉重来)

- 连续 3 个月夫妻有任一方未完成填报
- 妻子主动要求"用回 Excel 算了"
- 出现一次"数据丢失且备份恢复失败"的事故
- Dashboard XIRR 数字常被怀疑算错(信任崩塌)
- 月度填报实际平均时长 > 15 分钟

任一发生 → 暂停加新功能,做用户访谈 + 数据校验,先解决根因。
