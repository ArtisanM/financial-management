# 家庭账房 PRD — v0.2

> 已交付 v0.1(2026-05 阶段)。v0.2 在它之上做两件事:**账本视角增强** + 新增**「资产体检」模块**;不替换任何 v0.1 功能。

---

## 1. 版本目标

家庭已经在用 v0.1 后,会自然遇到四个新问题,v0.2 集中解决:

1. "我招行卡这一年里到底进出了多少笔?" — 当前 `/entry` 只看本期、`/reports/period/{id}` 看单期、`/admin/audit` 偏运维,**没有"单账户跨期完整流水"视图**
2. "我手快填错了一笔工资,怎么撤销?" — 当前只能 reopen 周期 + 覆盖余额绕一大圈
3. "在手机微信里点开链接 → 没法用 / 太丑;每次还要在 Safari 里输 URL" — 微信内浏览器限制 + 没 PWA
4. "我们家这套配置健康吗?这个账户跑得好不好?" — v0.1 只看资金进出,**没有理财表现 / 风险 / 建议视角**

v0.2 由两条主线组成:

**主线 A · 账本视角增强**(FR-30/31/32 + FR-33/34):
- FR-30 账户详情页(账本视角:余额时序 + 月分组流水)
- FR-31 单账户 CSV 导出 / FR-32 单条流水软删除
- FR-33 微信引导 / FR-34 iOS PWA(已交付 2026-05-09)

**主线 B · 新增「资产体检」模块**(FR-40 系列):
- 全家级诊断:配置 / 流动性 / 集中度 / 风险分布 / 加权年化
- 账户级诊断:类型差异化(投资/储蓄/负债/简版)+ 基准对照
- 智能建议引擎:10+ 条规则,全家级与账户级共用
- 产品类目 + 风险评级体系(数据基座)

**关注点分离原则** ★:
- 账户详情页 `/accounts/{id}` 只看**账本**(资金进出 / 余额 / 流水)
- 「资产体检」模块 `/checkup` 只看**诊断**(配置 / 风险 / 收益 / 建议)
- 账户页底栏有显眼入口「📊 看资产体检」→ `/checkup?account={id}` 自动按账户筛选
- 不在账户页保留任何"诊断 teaser"— 信息不分裂,用户心智清晰

**已交付(2026-05-09)**:FR-33 微信引导 + FR-34 iOS Safari PWA(含真机截图 modal + dev escape ?reset_pwa)+ 错误兜底页(/error 卡通页)+ 全局前端兜底(htmx:responseError/sendError/timeout + window error)+ entry 上期余额参考。共 93 条 QA 全 PASS。

> **核心愿景**:家庭账本不只是"看数据",而是"**给建议**"。
> 普通家庭(尤其非技术成员)不会自己消化 Sharpe / σ / 最大回撤这种术语,他们要的是 **"我们家这套配置健康吗?这个账户跑得好不好?我现在该做什么?"**。
>
> 这个新模块是 v0.2 区别于"账户列表 / 流水查看器"的核心价值。

**入口**:
- 主入口:**顶部 nav 新增「资产体检」**项(放在「报表」与「管理」之间)

### FR-40a · 全家级诊断 · P0

**位置**:`/checkup`(默认全家维度)— 顶部 KPI 之下、智能建议之前。

**4 张诊断卡(2x2 grid)**:

```
┌──────────────────────┬──────────────────────┐
│ 配置诊断              │ 风险分布              │
│  类型分布(环形图)     │  ★1-6 等级(环形)     │
│  CASH 32%             │  ★ 极低     38%       │
│  STOCK 28%            │  ★★ 低      24%       │
│  WEALTH 20%           │  ★★★ 中低   18%       │
│  LOAN -28%(负债)     │  ★★★★ 中    12%       │
│                       │  ★★★★★ 中高 8%        │
│  对比建议:            │  ★★★★★★ 极高 0%       │
│  「股票占比偏低,可    │                       │
│  考虑加配」            │  风险综合评分:中等    │
├──────────────────────┼──────────────────────┤
│ 流动性诊断            │ 收益诊断              │
│  紧急储备 14.2 月      │  全家加权年化 6.4%    │
│  健康线 6 月 ✓        │  ─                   │
│  现金占总资产 32%      │  vs 综合基准:         │
│  ─                   │   6.4% > 5.8% ✓ 跑赢  │
│  集中度告警:无       │  (按各账户类目        │
│                       │   benchmark_pct       │
│                       │   加权得到的综合)     │
└──────────────────────┴──────────────────────┘
```

**计算口径**:

- **类型分布** = Σ(同 type 账户余额 × FX 折 CNY)/ 全家总资产
- **风险分布** = Σ(同 risk_level 账户余额 × FX 折 CNY)/ 总资产(LOAN 不计入)
- **紧急储备月数** = Σ(CASH 类账户余额)/ 全家月均消费(近 12 月)
- **集中度** = max(单账户余额 / 总资产);> 40% 触发"过度集中"告警
- **全家加权年化** = Σ(账户当前价值 × 账户 XIRR)/ Σ(账户当前价值)
- **综合基准** = Σ(账户当前价值 × 类目 benchmark_pct)/ Σ(账户当前价值)
- **健康度评分(每账户)** = 100 - 命中规则的扣分(DANGER -25 / WARN -10 / INFO -3),用于账户列表排序

**验收**:
- [ ] 4 张卡渲染正确,数字与诊断逻辑一致
- [ ] LOAN 账户在配置诊断里以负值/单独色块显示,不影响"总资产"分母
- [ ] 紧急储备 < 3 月触发红色告警 banner
- [ ] 现金占比 > 50% 触发"现金堆积"提示
- [ ] 集中度 > 40% 触发告警

### FR-40b · 账户级类型差异化诊断 · P0

**位置**:`/checkup?account=X` — 账户标签栏之下、基准对照之前。

#### FR-40b-1 · 投资诊断(STOCK / WEALTH)

**布局**(左收益 + 右风险):

```
左栏:收益指标
  累计收益 +¥48,200 (+12.4%) · 累计净流入 ¥390,000 · 当前价值 ¥438,200
  年化 XIRR 6.8%(资金加权 24 月) · 年化 TWR 7.1%(剔除资金进出)
  净值曲线(起点 1.000)─────╲╱──── 最新 1.124

右栏:风险指标
  最大回撤 −8.3%(于 2025-10) · 回撤恢复 42 天
  年化波动率 σ = 11.2% · Sharpe 0.61(简化口径,无风险率=0%)
  月收益分布柱状(12 月)
```

**计算口径**:
- 当前价值 = 本期 period_snapshot.end_balance
- 累计净流入 = Σ(INCOME + TRANSFER_IN − EXPENSE − TRANSFER_OUT) 全部历史
- 累计收益 = 当前价值 − 累计净流入
- 年化 XIRR = `XirrCalculator.annualizedOrCumulative(...)`(复用 v0.1)
- 年化 TWR = 月度切片连乘 `Π(1 + r_i) − 1`(复用 v0.1 `TwrCalculator`)
- 净值曲线 = 月度切片连乘,起点 1.000
- 最大回撤 = `MaxDrawdownCalculator.compute(navSeries)`,**新增 calc 类**
- 月波动率(年化)= std(月收益率) × √12
- Sharpe = 年化收益率 / 年化波动率(简化口径)
- 回撤恢复天数 = 最大回撤底部到下次创新高的月份差 × 30(近似)

#### FR-40b-2 · 储蓄诊断(CASH)

**布局**(单栏 4 KPI 大数 + 月度收支柱):

```
紧急缓冲月数 ⏱ 14.2 月(健康 ≥ 6)
平均月入 ¥34,500 · 平均月出 ¥21,800
月均净储蓄 +¥12,700(净储蓄率 36.8%)
月度收支柱状(12 月)
```

**风险信号 banner**:
- 紧急缓冲 < 3 月 → 红色「缓冲不足 3 月,建议留 6 月以上」
- 单月最大支出 > 平均月出 × 3 → 黄色「最近某月异常大额支出」
- 连续 3 月净储蓄 < 0 → 红色「近 3 月持续超支」

### FR-40c · 智能建议引擎(全家级 + 账户级)· P0

> **产品愿景对齐**:从"消化 Sharpe/σ/回撤"→ 给具体行动建议。
> 全家级建议看到「家里整体配置 / 风险」;账户级建议看到「这个账户该做什么」。

**位置**:
- 全家级:`/checkup` 顶部 KPI 之下、账户列表之前
- 账户级:`/checkup?account=X` 类型诊断卡之后

**形态**(纸质便签卡片堆叠,每条建议一张):

```
┌──────────────────────────────────────────────────┐
│ 📌 来 · 自 · 账 · 房 · 的 · X · 条 · 提 · 醒     │
│                                                  │
│ ┌──────────────────────────────────────────────┐ │
│ │ ⚠️ 缓冲过厚                       〔流动性〕│ │
│ │ 招行储蓄卡-工资 余额 ¥73,200 ≈ 14 个月支出   │ │
│ │ — 已超过推荐的 6-9 月。                       │ │
│ │ 💡 把 ¥35,000 拨到「蚂蚁财富-基金」(年化     │ │
│ │    3.5% 稳健),预计每年多挣 ¥1,225。         │ │
│ │ [→ 现在划转]   [✕ 这条不适用]                │ │
│ └──────────────────────────────────────────────┘ │
│ ...                                              │
└──────────────────────────────────────────────────┘
```

**生成机制**:**纯 Java 规则引擎**,非 LLM,完全可预测。每次访问 `/checkup` 实时跑(< 50ms)。

```java
record Advice(
    AdviceLevel level,       // OK / INFO / WARN / DANGER
    AdviceCategory category, // LIQUIDITY / EFFICIENCY / RETURN_QUALITY / CONCENTRATION / PROGRESS / BEHAVIOR / ALLOCATION
    AdviceScope scope,       // FAMILY / ACCOUNT
    Long accountId,          // null = 全家级
    String headline, body, suggestion,
    Optional<Action> action  // 一键操作(可选)
) {}
```

### FR-32 · 单条流水软删除 · P0

**触发**:每条流水末尾 `⋮` 菜单 → "删除此条"(二次 confirm)

**约束**:
- 周期 status=CLOSED 的流水**不可删**(POST 直接 toast "周期已关闭,需先重新打开")
- LOAN 账户的"自动 prefill 草稿 transfer"**不可删**(因为是系统生成的,改 LOAN 余额时再处理)

**端点**:
- `POST /entry/cash-flow/{cfId}/delete` — 软删 cash_flow
- `POST /entry/transfer/{tId}/delete` — 软删 transfer
- `POST /entry/snapshot/{periodId}/{accountId}/delete` — **不支持**(snapshot 是余额权威,只能 reopen 周期后重填)

**数据模型**:
```sql
ALTER TABLE cash_flow ADD COLUMN deleted_at TIMESTAMP(3) NULL;
ALTER TABLE transfer  ADD COLUMN deleted_at TIMESTAMP(3) NULL;
CREATE INDEX idx_cash_flow_active ON cash_flow(period_id, account_id, deleted_at);
CREATE INDEX idx_transfer_active ON transfer(period_id, deleted_at);
```

**所有既有 mapper 查询**加 `WHERE deleted_at IS NULL` 过滤。

**软删后的副作用**:
- snapshot.end_balance **不动**(用户手动决定的权威值)
- 由此产生的差额会出现在该账户的"未解释金额"上 → 用户看到后可手动校准
- 审计:`audit_log type=CASH_FLOW_DELETE / TRANSFER_DELETE`,含 `payload_json`(流水原值)

**验收**:
- [ ] 删除 OPEN 期流水成功;DB 中 `deleted_at` 写入,UI 该条消失
- [ ] 删除 CLOSED 期流水触发 toast 拒
- [ ] 删除后:本账户"收入/支出/转入/转出"4 字段汇总变化、未解释金额相应增加 / 减少
- [ ] 审计日志含被删除流水的 amount/kind/note,可恢复
