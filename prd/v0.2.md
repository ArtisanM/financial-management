# 家庭账房 PRD — v0.2

> 已交付 v0.1(2026-05 阶段)。v0.2 在它之上做两件事:**账本视角增强** + 新增**「资产体检」模块**;不替换任何 v0.1 功能。

---

## 1. 版本目标

家庭已经在用 v0.1 后,会自然遇到四个新问题,v0.2 集中解决:

1. "我招行卡这一年里到底进出了多少笔?" — 当前 `/entry` 只看本期、`/reports/period/{id}` 看单期、`/admin/audit` 偏运维,**没有"单账户跨期完整流水"视图**
2. "我手快填错了一笔工资,怎么撤销?" — 当前只能 reopen 周期 + 覆盖余额绕一大圈
3. "在手机微信里点开链接 → 没法用 / 太丑;每次还要在 Safari 里输 URL" — 微信内浏览器限制 + 没 PWA
4. "我们家这套配置健康吗?这个账户跑得好不好?" — v0.1 只看资金进出,**没有理财表现 / 风险 / 建议视角**

v0.2 由两条主线组成:

**主线 A · 账本视角增强**(FR-30/31/32 + FR-33/34):
- FR-30 账户详情页(账本视角:余额时序 + 月分组流水)
- FR-31 单账户 CSV 导出 / FR-32 单条流水软删除
- FR-33 微信引导 / FR-34 iOS PWA(已交付 2026-05-09)

**主线 B · 新增「资产体检」模块**(FR-40 系列):
- 全家级诊断:配置 / 流动性 / 集中度 / 风险分布 / 加权年化
- 账户级诊断:类型差异化(投资/储蓄/负债/简版)+ 基准对照
- 智能建议引擎:10+ 条规则,全家级与账户级共用
- 产品类目 + 风险评级体系(数据基座)

**关注点分离原则** ★:
- 账户详情页 `/accounts/{id}` 只看**账本**(资金进出 / 余额 / 流水)
- 「资产体检」模块 `/checkup` 只看**诊断**(配置 / 风险 / 收益 / 建议)
- 账户页底栏有显眼入口「📊 看资产体检」→ `/checkup?account={id}` 自动按账户筛选
- 不在账户页保留任何"诊断 teaser"— 信息不分裂,用户心智清晰

**已交付(2026-05-09)**:FR-33 微信引导 + FR-34 iOS Safari PWA(含真机截图 modal + dev escape ?reset_pwa)+ 错误兜底页(/error 卡通页)+ 全局前端兜底(htmx:responseError/sendError/timeout + window error)+ entry 上期余额参考。共 93 条 QA 全 PASS。

### FR-40 · 「资产体检」模块 · `/checkup` · P0

> **核心愿景**:家庭账本不只是"看数据",而是"**给建议**"。
> 普通家庭(尤其非技术成员)不会自己消化 Sharpe / σ / 最大回撤这种术语,他们要的是 **"我们家这套配置健康吗?这个账户跑得好不好?我现在该做什么?"**。
>
> 这个新模块是 v0.2 区别于"账户列表 / 流水查看器"的核心价值。

**入口**:
- 主入口:**顶部 nav 新增「资产体检」**项(放在「报表」与「管理」之间)

#### FR-40 · 四大核心维度选择 ★

> 体检模块"看什么"的产品决策。家庭理财体检可考虑的维度有 10+ 种,v0.2 收敛到 4 个,其余留 v0.3+。

**候选 10 个维度**(按"家庭实际关心度"排序):

| # | 维度 | 关心度 | 数据成熟度 | v0.2 决定 |
|---|---|---|---|---|
| 1 | **流动性 / 应急储备** | ★★★★★ | ✅ 已有 | **选定** |
| 2 | **风险敞口 + 配置均衡 + 集中度** | ★★★★★ | ✅ 已有(融合 #5)| **选定** |
| 3 | **收益质量(vs 基准)** | ★★★★★ | ✅ 已有 | **选定** |
| 4 | **负债健康** | ★★★★★ | ✅ 已有(LOAN 时序)| **选定** |
| 5 | 集中度 | ★★★★ | ✅ | 融入 #2 |
| 6 | 储蓄率 | ★★★ | ⚠ 数据需 12+ 月 | 搁置 |
| 7 | 现金流稳定性 | ★★★ | ⚠ 假阳性高 | 搁置 |
| 8 | 历史趋势(净资产 / 加权年化时序) | ★★★ | ✅ | dashboard 已部分覆盖 |
| 9 | 币种分散 | ★★ | ⚠ 数据少 | 搁置 |
| 10 | 税务 / 成本效率 | ★★ | ⚠ 中国场景弱 | 搁置 |

**选定 4 个核心维度**:

```
1. 流动性(Liquidity)
   ├─ 应急储备月数 / 现金占比
   ├─ 工程算 → 规则:LIQ-1/2 / EFF-1 / FAM-LIQ-1
   ├─ 流动资产 = LIQUID 类账户合计(CASH + 货币基金等 product_category.liquidity_class=LIQUID)
   │  · 2026-05-14 修订:WEALTH 账户内 MONEY_FUND(余额宝/零钱通)正确归入流动资产
   │    (前置 bug:仅按 AccountType 一刀切,货币基金被误归 SEMI_LIQUID,流动资产被低估)
   └─ 全家诊断卡:流动性诊断

2. 风险与配置(Risk & Allocation)
   ├─ 风险等级分布 / 单账户单类目集中度 / 整体偏离度
   ├─ 工程算 → 规则:RISK-1/2 / FAM-CON-1/2 / FAM-RISK-1/2
   └─ 全家诊断卡:配置诊断 + 风险分布(共 2 张)

3. 收益质量(Return Quality)
   ├─ 年化(XIRR / TWR)/ Sharpe / 最大回撤 / vs 基准
   ├─ 工程算 → 规则:RET-1/2/3/4 / FAM-ALC-1/2
   └─ 全家诊断卡:收益诊断 + 账户卡:投资诊断卡

4. 负债健康(Debt Health)
   ├─ 还款进度 / 月均还款 / 利息成本 / 负债率
   ├─ 工程算 → 规则:PRG-1/2
   └─ 账户卡:负债进度卡(LOAN 类型)
```

**为什么是 4 不是 10**:
- 体检报告的可读性:4 个维度形成"完整诊断叙事"(应急 → 风险 → 收益 → 负债),用户能一眼读完;10 个维度反而稀释信号
- 数据成熟度:储蓄率 / 现金流稳定性需要 12+ 月数据,v0.2 阶段用户填报历史不足
- 搁置维度多数有"假阳性高 / 中国场景弱 / 已有别处覆盖" 三类问题,v0.2 实施只增加噪音
- 4 个维度对应 PRD 已设计的 18 条规则 + FR-40a 4 张诊断卡 + FR-40b 类型差异化 — 自洽闭环

**v0.3+ 评估增加**:
- 储蓄率(用户用满 12 月后启用)
- 历史趋势独立模块(同比 / 环比 / 长期复利推演 — 见 v0.1 §7.6 FR-23~33 远期)

### FR-40a · 全家级诊断 · P0

**位置**:`/checkup`(默认全家维度)— 顶部 KPI 之下、智能建议之前。

**4 张诊断卡(2x2 grid)**:

```
┌──────────────────────┬──────────────────────┐
│ 配置诊断              │ 风险分布              │
│  类型分布(环形图)     │  ★1-6 等级(环形)     │
│  CASH 32%             │  ★ 极低     38%       │
│  STOCK 28%            │  ★★ 低      24%       │
│  WEALTH 20%           │  ★★★ 中低   18%       │
│  LOAN -28%(负债)     │  ★★★★ 中    12%       │
│                       │  ★★★★★ 中高 8%        │
│  对比建议:            │  ★★★★★★ 极高 0%       │
│  「股票占比偏低,可    │                       │
│  考虑加配」            │  风险综合评分:中等    │
├──────────────────────┼──────────────────────┤
│ 流动性诊断            │ 收益诊断              │
│  紧急储备 14.2 月      │  全家加权年化 6.4%    │
│  健康线 6 月 ✓        │  ─                   │
│  现金占总资产 32%      │  vs 综合基准:         │
│  ─                   │   6.4% > 5.8% ✓ 跑赢  │
│  集中度告警:无       │  (按各账户类目        │
│                       │   benchmark_pct       │
│                       │   加权得到的综合)     │
└──────────────────────┴──────────────────────┘
```

**计算口径**:

- **类型分布** = Σ(同 type 账户余额 × FX 折 CNY)/ 全家总资产
- **风险分布** = Σ(同 risk_level 账户余额 × FX 折 CNY)/ 总资产(LOAN 不计入)
- **紧急储备月数** = Σ(CASH 类账户余额)/ 全家月均消费(近 12 月)
- **集中度** = max(单账户余额 / 总资产);> 40% 触发"过度集中"告警
- **全家加权年化** = Σ(账户当前价值 × 账户 XIRR)/ Σ(账户当前价值)
- **综合基准** = Σ(账户当前价值 × 类目 benchmark_pct)/ Σ(账户当前价值)
- **健康度评分(每账户)** = 100 - 命中规则的扣分(DANGER -25 / WARN -10 / INFO -3),用于账户列表排序

**验收**:
- [ ] 4 张卡渲染正确,数字与诊断逻辑一致
- [ ] LOAN 账户在配置诊断里以负值/单独色块显示,不影响"总资产"分母
- [ ] 紧急储备 < 3 月触发红色告警 banner
- [ ] 现金占比 > 50% 触发"现金堆积"提示
- [ ] 集中度 > 40% 触发告警

### FR-40b · 账户级类型差异化诊断 · P0

**位置**:`/checkup?account=X` — 账户标签栏之下、基准对照之前。

#### FR-40b-1 · 投资诊断(STOCK / WEALTH)

**布局**(左收益 + 右风险):

```
左栏:收益指标
  累计收益 +¥48,200 (+12.4%) · 累计净流入 ¥390,000 · 当前价值 ¥438,200
  年化 XIRR 6.8%(资金加权 24 月) · 年化 TWR 7.1%(剔除资金进出)
  净值曲线(起点 1.000)─────╲╱──── 最新 1.124

右栏:风险指标
  最大回撤 −8.3%(于 2025-10) · 回撤恢复 42 天
  年化波动率 σ = 11.2% · Sharpe 0.61(简化口径,无风险率=0%)
  月收益分布柱状(12 月)
```

**计算口径**:
- 当前价值 = 本期 period_snapshot.end_balance
- 累计净流入 = Σ(INCOME + TRANSFER_IN − EXPENSE − TRANSFER_OUT) 全部历史
- 累计收益 = 当前价值 − 累计净流入
- 年化 XIRR = `XirrCalculator.annualizedOrCumulative(...)`(复用 v0.1)
- 年化 TWR = 月度切片连乘 `Π(1 + r_i) − 1`(复用 v0.1 `TwrCalculator`)
- 净值曲线 = 月度切片连乘,起点 1.000
- 最大回撤 = `MaxDrawdownCalculator.compute(navSeries)`,**新增 calc 类**
- 月波动率(年化)= std(月收益率) × √12
- Sharpe = 年化收益率 / 年化波动率(简化口径)
- 回撤恢复天数 = 最大回撤底部到下次创新高的月份差 × 30(近似)

#### FR-40c · AI 智能诊断层 ★(2026-05-10 修订:从"文案润色"升级为"综合智能诊断")

> **核心产品决策(2026-05-10 修订)**:智能建议不是"规则给硬骨架 + LLM 换语气",而是 **"规则给硬底,AI 在硬底之上做综合判断 + 跨规则推理"** 的两层架构。
>
> **旧方向(已废弃)**:60% 规则 + 40% LLM 文案润色。LLM 受 prompt + Validator 双重锁,只能换词,不能引入新数字、新结论、新建议。
>
> **新方向(2026-05-10 起)**:60% 规则硬底 + 40% AI 智能诊断。AI 看到全家完整画像(所有命中规则 + 全家级聚合数字 + 各账户硬事实),做**跨规则、跨账户、综合性**的判断和建议。

**为什么从"润色"升级到"智能诊断"**:

| 路线 | 决策 | 理由 |
|---|---|---|
| 纯规则模板填空 | ❌ 早期否决 | 文案僵硬,无综合判断,跨规则无能力 |
| 60% 规则 + 40% 润色(旧方向) | ❌ 2026-05-10 废弃 | 受限太死,LLM 价值仅停留在"换词";用户感知不到"AI 在帮我诊断" |
| **60% 规则硬底 + 40% AI 综合诊断**(2026-05-10 新选定) | ✅ | 规则保证数字硬,AI 看到全局做判断;用户在 /checkup 看到"工程严密的硬数据 + 顾问视角的综合诊断"两层 |
| 纯 LLM 生成 | ❌ | 数字幻觉风险高,审计差,不可重现;但可以**给 AI 看到完整画像**而不必走"纯 LLM" |

**两层架构(2026-05-10 新)**:

```
第一层 · 规则引擎硬底(继续保留,不变)
  ├─ 工程算法精确触发 16 条规则(10 账户 + 6 家庭)
  ├─ 输出便签卡片堆叠(DANGER/WARN/INFO/OK)
  ├─ 每条卡显示规则 ID + 硬数字 + 模板文案
  └─ 用户可一目了然看到"系统精准识别的问题"

第二层 · AI 智能诊断(2026-05-10 新增)
  ├─ 在便签卡组下方,新增「来自资产顾问的综合诊断」长文区块
  ├─ 进 /checkup 页面时,异步 fetch + spinner placeholder + fade-in 替换
  ├─ AI 看到:全家完整快照(账户名 / 主理人 / 类目 / 风险 / 余额 / 占比 /
  │           年化 / 加权综合 / 命中规则 ID 列表 / 各规则硬事实)
  ├─ AI 自由综合判断:跨规则推理("LOAN 占比 28% + STOCK 30% 下,应急 14 月反而合理")、
  │                   跨账户建议("把多出的 ¥35k 放短债基金而非货基,多赚 1pp")、
  │                   规则没覆盖的视角("整体偏稳健,但缺乏中长期资产")
  ├─ 失败时:显示「AI 暂时不可用,以下为规则硬数据」占位文案,不静默隐藏
  └─ AI 输出 200-500 字综合诊断,允许引入新数字 / 新建议
```

**工程层 vs AI 层 责任边界(新)**:

```
工程层(规则引擎 + Calculator)负责:
  ✓ 所有数学计算(XIRR / TWR / σ / Sharpe / MaxDD / 加权)
  ✓ 阈值判断 + 规则触发 + Level 决定
  ✓ 推荐金额估算 / 预期收益计算
  ✓ 输出便签卡(永远在第一层,UI 永远显示)

AI 层(Qwen-Plus 主 / DeepSeek-Chat 备)负责:
  ✓ 看到全家硬事实 + 命中规则集合,做综合判断
  ✓ 跨规则推理:把 5 条单独规则结合起来给整体视角
  ✓ 跨账户建议:推荐"从 A 账户调到 B 账户"这种工程算不出来的搭配
  ✓ 给规则没覆盖的视角(年龄阶段、生活阶段、家庭结构推断)
  ✓ 现代投资顾问语调 + 不啰嗦 + 200-500 字综合段落
  ✓ 允许引入合理数字(如 "1pp 年化"、"短债 vs 货基差 0.5%")

AI 层不能做(prompt + Validator 兜底):
  ✗ 编造**金额**(出现 ¥xxx 数字必须能在硬事实里找到来源,或合理推导)
  ✗ 给具体产品名(余额宝 / 510300 / 茅台 / 招商银行某某理财产品)

  ✗ 给金融建议背书("保证 X% 收益")

OutputValidator 兜底(简化版):
  ✓ 禁词扫描(具体产品名 + 古典中式词 + 担保性话术)
  ✓ 长度校验(150-600 字)
  ✓ 不再"锁原文每个数字必须保留"(允许 AI 引入推理性新数字)
  ✓ 任一不通过 → 显示「AI 暂时不可用」占位文案 + 写 audit_log LLM_REJECTED
```

**为什么这样划界**:
- 用户(家庭管理员)显式承诺"家庭资产数据(账户/流水/指标/家庭名)不需脱敏",这是个人决策权。新方向 AI 综合诊断需要看到完整画像才能做跨账户推理,任何不必要的人为脱敏都会削弱诊断质量。
- **唯一例外是主理人真名**:即使家庭名/账户名都上传了,自然人姓名仍是最高敏感的 PII,留在本地。后端 `PromptBuilder` 在组装 prompt 前,把所有 member 真名按 id 升序稳定映射为「成员A / 成员B / 成员C」,映射只在内存里,不入 LLM。AI 输出涉及成员代号时,前端渲染时反映射回真名再显示给用户(用户体验上感觉不到映射)。

**实现要点**:
- 映射规则:family.members 按 `id ASC` 排序,第 1 位 → 成员A,第 2 位 → 成员B,以此类推。同一家庭的会话里映射稳定,LLM 输出"成员A 应增加现金储备"时前端能正确反映射回"Alice"。
- LLM 输出含真名(不应该,但万一):Validator 扫描原始 member 真名,命中即降级 + audit_log。这是双保险,理论上 LLM 看不到真名就不会写出来,但作为防御深度保留。

### FR-32 · 单条流水软删除 · P0

**触发**:每条流水末尾 `⋮` 菜单 → "删除此条"(二次 confirm)

**约束**:
- 周期 status=CLOSED 的流水**不可删**(POST 直接 toast "周期已关闭,需先重新打开")
- LOAN 账户的"自动 prefill 草稿 transfer"**不可删**(因为是系统生成的,改 LOAN 余额时再处理)

**端点**:
- `POST /entry/cash-flow/{cfId}/delete` — 软删 cash_flow
- `POST /entry/transfer/{tId}/delete` — 软删 transfer
- `POST /entry/snapshot/{periodId}/{accountId}/delete` — **不支持**(snapshot 是余额权威,只能 reopen 周期后重填)

**数据模型**:
```sql
ALTER TABLE cash_flow ADD COLUMN deleted_at TIMESTAMP(3) NULL;
ALTER TABLE transfer  ADD COLUMN deleted_at TIMESTAMP(3) NULL;
CREATE INDEX idx_cash_flow_active ON cash_flow(period_id, account_id, deleted_at);
CREATE INDEX idx_transfer_active ON transfer(period_id, deleted_at);
```

**所有既有 mapper 查询**加 `WHERE deleted_at IS NULL` 过滤。

**软删后的副作用**:
- snapshot.end_balance **不动**(用户手动决定的权威值)
- 由此产生的差额会出现在该账户的"未解释金额"上 → 用户看到后可手动校准
- 审计:`audit_log type=CASH_FLOW_DELETE / TRANSFER_DELETE`,含 `payload_json`(流水原值)

**验收**:
- [ ] 删除 OPEN 期流水成功;DB 中 `deleted_at` 写入,UI 该条消失
- [ ] 删除 CLOSED 期流水触发 toast 拒
- [ ] 删除后:本账户"收入/支出/转入/转出"4 字段汇总变化、未解释金额相应增加 / 减少
- [ ] 审计日志含被删除流水的 amount/kind/note,可恢复

```sql
CREATE INDEX idx_cf_deleted ON cash_flow(deleted_at);
CREATE INDEX idx_tr_deleted ON transfer(deleted_at);

-- V11__step_v02_product_category.sql · 产品类目 + 风险评级
CREATE TABLE product_category (
  code             VARCHAR(40) PRIMARY KEY,
  display_name     VARCHAR(80)  NOT NULL,
  risk_level       TINYINT      NOT NULL,
  benchmark_label  VARCHAR(80)  NULL,
  benchmark_pct    DECIMAL(5,2) NULL,
  applicable_types VARCHAR(80)  NOT NULL,
  description      VARCHAR(255) NULL,
  display_order    SMALLINT NOT NULL DEFAULT 100
);
ALTER TABLE account
  ADD COLUMN product_category_code VARCHAR(40) NULL,
  ADD COLUMN risk_level_override   TINYINT     NULL,
  ADD CONSTRAINT fk_account_product_category
    FOREIGN KEY (product_category_code) REFERENCES product_category(code);
CREATE INDEX idx_account_product_category ON account(product_category_code);
-- 同迁移内 INSERT 16 行预置类目 + UPDATE 已有 account 回填默认 category
```

所有既有 mapper SELECT 加 `AND deleted_at IS NULL`(用 `<sql>` 片段统一注入或显式改写)。
