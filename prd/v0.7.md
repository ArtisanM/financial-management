# PRD · v0.7 · 一键 Docker 部署 + 兼容存量用户

> 状态:**草案 · 待评审**(按 [[feedback_design_process]],PRD → 你评审 → preview → tech-design → 代码)

## 0. 背景

开源宣传铺开后,会来一批想自部署的人。现有 `deploy/deploy.sh` 是「在服务器上直装 JDK/Maven/MySQL/nginx + 现编 jar」的重路径:污染宿主机、装依赖慢、出过依赖地狱(上次 certbot 的 cryptography 冲突)、只认 apt/dnf+systemd、升级要重编。门槛高 = 漏转化。

v0.7 目标:**`docker compose up -d` 一条命令起全套**,干净、跨平台、可跑在任何 Docker 主机/NAS;同时**绝不破坏已 systemd 部署的存量用户(包括我们自己的 prod + beta)**,并给他们一条平滑迁到 Docker 的路。

## 1. 目标用户 & 场景

- **新用户**:有一台能跑 Docker 的机器(VPS/NAS/本地),想几分钟跑起来,不想在宿主机装一堆东西。
- **存量 systemd 用户(含我们自己)**:已用 deploy.sh 跑着,数据在宿主机 MySQL + uploads + finance.env。希望:① 现状不受任何影响;② 想迁 Docker 时有一键迁移、数据零丢。

## 2. 非目标(本版不做)

- 不做 k8s / helm / swarm(单机 compose 足够,过度工程)
- 不强制存量用户迁移(systemd 路径继续支持,不弃)
- 不重写迁移机制(继续用 `db/apply.sh` + schema_history,容器内复用)

## 3. 功能需求(FR)

**FR-120 一键 compose 起全套** —— `docker compose up -d` 起 app + MySQL(可选反代),自动建库、跑迁移、起服务;`docker compose down` 干净停;数据留存。

**FR-121 数据持久化** —— 命名卷:`db-data`(MySQL)/ `uploads`(logo)/ `backups`(备份)。**容器删了、重建,数据不丢**。

**FR-122 配置与密钥** —— 一个 `.env`(从 `.env.example` 拷)承载 DB 密码、`REMEMBER_ME_KEY`、端口、时区等;首次可脚本**自动生成随机 DB 密码 + REMEMBER_ME_KEY**。LLM key / 阈值 / 短信 aksk 等运营参数**仍走管理页(DB `family_runtime_config`)**,不进 compose env(承 [[feedback_admin_runtime_config]])。密钥**不进镜像、不进日志明文**。

**FR-123 启动即迁移** —— app 容器 entrypoint:等 MySQL healthcheck 通过 → 跑 `db/apply.sh`(版本化迁移)→ 启动 Spring。**与 systemd 路径共用同一套 `V*.sql` + `schema_history`**,幂等,从 systemd 迁来的库不会重放。

**FR-124 反代 / HTTPS 交给用户(compose 保持最小)** —— compose **只起 app + MySQL**,app 暴露端口(默认 20000),**不内置反代、不内置 HTTPS**。域名/HTTPS 由用户在前面自己挂:已有 nginx 的把 `proxy_pass` 指到容器端口即可;文档给 **nginx+certbot** 和 **Caddy** 两段示例片段供照抄。理由:兼容存量 nginx 用户、不替人做反代决策、compose 最小最干净。

**FR-125 镜像分发(双路)** —— ① **预构建镜像**推到 GHCR,`docker compose pull` 秒拉、免宿主机编译;② 保留 **Dockerfile**(多阶段:maven 构建 → JRE 运行)供源码自构建。文档给**国内镜像加速**指引(GHCR/DockerHub 在大陆慢)。

**FR-126 向后兼容 + 迁移** —— ① systemd `deploy.sh` 路径**原样保留**,存量用户零改动;② 新增 **`deploy/migrate-to-docker.sh`**:对一台现有 systemd 机器,自动 `mysqldump` 备份 → 灌进 Docker 的 `db-data` 卷 → 搬 `uploads` → 由 `/etc/finance.env` 生成 compose `.env` → `compose up`。因共用 `schema_history`,迁移后**不重放迁移、版本状态完整**。

**FR-127 备份平价** —— Docker 下保留每日 `mysqldump` 到 `backups` 卷 + 保留天数(sidecar cron 容器或 app 内调度),与现有 systemd timer 行为对齐。

**FR-128 升级** —— `docker compose pull && docker compose up -d`:拉新镜像 → entrypoint 自动跑增量迁移 → 滚动重启。失败不毁数据(迁移幂等 + 升级前自动 dump)。

## 4. 关键决策(**已定 · 2026-06-11**)

- **决策 A · 镜像分发 = 双路**:发布**预构建镜像到 GHCR**(`pull` 秒拉、免宿主机编译)+ 保留 **Dockerfile** 供源码自构建 + 文档教国内镜像加速。默认 compose 用预构建镜像,可切源码构建。→ 需加一条 GitHub Actions 发布流水线(打 tag 时 build+push 多架构镜像)。
- **决策 B · compose 不带反代/HTTPS**:compose **只 app + MySQL**,反代/HTTPS 完全交给用户(见 FR-124)。不内置 Caddy。
- **决策 C · 两路并存 + 迁移脚本**:systemd `deploy.sh` 路径原样保留不弃(我们 prod/beta 也是存量,不冒险);Docker 作新推荐路径;另给一键 `migrate-to-docker.sh`(自动备份 + 灌数据 + 搬 uploads),想迁的人可选,数据零丢、版本不重放。

## 5. 红线 / backward-compat

- **存量 systemd 用户(含 prod 39.107 + beta)零破坏** —— 这是最高红线,Docker 是「新增」不是「替换」。
- 迁移前**强制自动备份**(mysqldump),失败可回到 systemd。
- **共用 `schema_history`** 防迁移后重放;`V*.sql` 一套两用(systemd + Docker)。
- 密钥(DB 密码 / REMEMBER_ME_KEY / LLM key / aksk)**不烤进镜像、不进日志**;运营参数仍走管理页 DB(承既有私密红线)。
- 镜像内不带任何家庭数据;`.env` git-ignored。

## 6. 验收基线

1. **全新机**:装好 Docker → `cp .env.example .env`(或跑 init 脚本)→ `docker compose up -d` → 浏览器登录、填报、AI 体检全通。
2. **反代/HTTPS 文档**:照文档的 nginx 或 Caddy 片段把已有反代指到容器端口 → 域名 HTTPS 通(compose 本身不负责)。
3. **存量迁移**:拿一台现有 systemd 部署(可用 beta 演练)跑 `migrate-to-docker.sh` → 数据(账户/周期/上传 logo)**零丢**,版本不重放。
4. **存量不动**:旧 `deploy.sh` 迭代路径仍可用(prod 继续 systemd 不受影响)。
5. **升级**:`compose pull && up -d` 自动增量迁移,数据保留。
6. mvn test 全绿 + qa-run 加 v07-DOCKER-* 守护(镜像构建成功 / 迁移幂等 / 卷持久化)。

## 7. 分期(供评审)

- **P1**:Dockerfile + compose(app+MySQL)+ entrypoint 迁移 + .env + 数据卷 + 文档 → 跑通「全新机一键起」
- **P2**:`migrate-to-docker.sh`(存量迁移)+ 备份 sidecar
- **P3**:GHCR 预构建镜像发布流水线(GitHub Actions)+ 反代/HTTPS 文档片段(nginx/Caddy)
- 也可一次性做完,分期不影响 PRD 完整性。

# v0.7 第二批 · 外部服务配置引导(易用性改造)

> 状态:**草案 · 待评审**(2026-06-16)· 决策已与用户确认,见下 §8.4

## 8.0 背景

一键 Docker 解决了「跑起来」,但跑起来之后,用户要接的外部服务(Qwen 百炼 key、DeepSeek key、阿里云短信 aksk/签名/模板)**没有好的配置引导**:
- 现状盘点:`docs/` 下其实已有 `llm-api-keys-setup.md`(195 行)、`aliyun-sms-setup.md`(207 行)、`llm-vendor-comparison.md`,但 **README 几乎没链**,新用户找不到;
- 系统内:短信页已有字段提示 + **一键测试短信**;但 **LLM 配置页(`integrations.html`)只有一行极简提示,没有申请步骤、没有「测试连接」按钮、没链到文档**。

开源用户拿到的体验是:看到几个 key 输入框,不知道去哪申请、填完不知道对不对。本批就补这块。

## 8.1 目标用户 & 场景

- 刚把容器跑起来的自部署用户,想开 AI 体检 / 短信提醒,面对空的 key 输入框不知所措。
- 目标:**不跳出系统就能知道「这个 key 干嘛的、去哪弄、填得对不对」**;README 侧有一个总入口能一站式查所有外部接入。

## 8.2 非目标

- 不做账号代申请 / OAuth 托管(就是引导 + 链接,key 仍用户自己去平台申请)。
- 不改运营参数仍走管理页 DB 的架构(承 [[feedback_admin_runtime_config]])。
- 不动私密红线:key 不进 prompt / 日志明文 / 前端明文回显。

## 8.3 功能需求(FR)

**FR-129 配置总指南文档 + README 入口** —— 新建 `docs/configuration.md`(配置与接入总指南):开篇讲清「**所有外部服务都可选,核心功能零配置可跑;配了各自解锁什么**」+ 一张总表(配置项 / 解锁什么功能 / 是否必需 / 去哪配[管理页路径] / 申请步骤摘要);每项摘要步骤 + 链到现有详版(`llm-api-keys-setup.md`、`aliyun-sms-setup.md`、`llm-vendor-comparison.md`)。README 加一个**醒目入口**指向它。

**FR-130 管理页内嵌折叠式申请指引** —— LLM 配置页(`integrations.html`)每个 key 字段旁加**折叠式帮助**(默认收起,点「如何获取?」展开):3–4 步申请步骤 + 平台控制台直链(百炼 / platform.deepseek.com)+ 「详见配置指南」链到 `docs/configuration.md`。短信页(`notification.html`)已有提示,补一个「详见配置指南」链接。形式用折叠 `<details>`,不撑长页面。

**FR-131 LLM 一键测试连接** —— LLM 配置页新增「测试连接」按钮(Qwen / DeepSeek 各一,或共用):点击后后端用当前保存的 key 发一个**最小开销**的探测请求,返回**成功 / 失败 + 脱敏错误原因**(如 401 key 无效 / 余额不足 / 网络不通)。补齐「外部接入必配一键测试」纪律(短信已有,LLM 缺)。

**FR-132 「可选 + 解锁什么」状态说明** —— LLM 配置页顶部一句话:**「这些都可选;不配则 AI 资产体检 / 调仓建议不可用,其余功能正常」**;短信页同理标注「不配则只走站内 banner 提醒,不影响记账」。让用户清楚「不配会怎样、配了得到什么」。

## 8.4 关键决策(**已与用户确认 · 2026-06-16**)

- **指引形式 = 折叠式内嵌步骤**(非「仅一行外链跳出」、非「永久铺开撑长页」):字段旁 `<details>`「如何获取?」,页面不胀又不用离开。
- **本批做 LLM「测试连接」按钮**(短信已有测试,LLM 这次补齐)。
- **文档 = 建 `docs/configuration.md` 总入口**(非「README 直接列 3 条链接」):一个总指南 + 总表,现有 3 篇详版作为子文档链进去,README 只需链这一个。

## 8.5 红线 / backward-compat

- **私密红线不破**:测试连接时 key 不回显明文、不写进 audit_log 明文、错误信息脱敏(不回显 key 本身);承既有 `PrivacyIsolationTest` + `v04-PRIV-1` 纪律。
- 测试请求要**最小开销**(别烧用户额度 / 别误触发计费),且失败要可读(区分 key 错 / 欠费 / 网络)。
- 纯增量:不改 `family_runtime_config` schema、不动现有配置读取链(DB > env > 默认);对已配好的存量用户零影响。
- UI 不用 emoji,用项目自带 inline SVG(承 [[feedback_no_emoji]]);面向用户文案避免技术黑话(承 [[feedback_user_friendly_naming]])。

## 8.6 验收基线

1. README 有醒目「配置与接入」入口 → 点进 `docs/configuration.md` → 总表能一眼看清 3 个外部服务各干嘛、去哪配、是否必需。
2. LLM 配置页:每个 key 旁能展开看到申请步骤 + 控制台直链 + 文档链;顶部有「可选 + 解锁什么」说明。
3. LLM「测试连接」:填对的 key → 返回成功;填错/留空 → 返回脱敏的失败原因;全程 key 不明文回显、不进日志。
4. 短信页有「详见配置指南」链接;原有一键测试短信不受影响。
5. 私密回归:`PrivacyIsolationTest` + `v04-PRIV-1` 仍绿;新增 qa-run 守护(测试连接端点存在 + 错误脱敏 + 文档/入口齐)。
6. mvn test 全绿。

## 8.7 分期

- **P1**:`docs/configuration.md` 总指南 + README 入口(纯文档,先让用户有处可查)。
- **P2**:管理页折叠式指引(LLM + 短信链接)+ 顶部「可选/解锁」说明。
- **P3**:LLM「测试连接」按钮 + 后端探测端点(脱敏错误)。
- 可一次性做完。

# 九、第三批 · 系统内首次引导(易用性改造)

> 状态:**草案 · 待评审**(2026-06-16)· 承本批易用性主线(文档侧已落地:README 上手 section + FAQ)

## 9.0 背景

部署成功 ≠ 会用。新用户第一次登录,面对的是一个**空仪表盘**(无 OPEN 周期、无自建账户、demo 数据已清),系统里**没有任何「下一步做什么」的引导**。README 虽写了 6 步,但产品内零提示——「部署成功却不会用」是仅次于「装不上」的流失点。把「怎么开始用」做进产品本身。

## 9.1 目标用户 & 场景

刚部署完、第一次登录的自托管用户(或刚清空重来的)。目标:**登进来就知道「先干什么、点哪里」**,不用回去翻 README。

## 9.2 非目标

- 不做打断式弹窗 / 多步 modal 向导 / 交互式 page tour(过度工程、移动端体验差)。
- 不改核心数据模型、不改周期/账户/填报的既有逻辑。
- 不做「示例数据一键填充」(易和真实数据混淆,违背隐私初衷)。

## 9.3 功能需求(FR)

**FR-133 首次引导面板(空状态)** —— 落地页(`home` / dashboard)在「**家庭尚未完成初始化**」时,在最显眼处显示一块**引导面板**:用一句话讲清核心流程,列出起步步骤,每步是**直达按钮**(点了跳对应页)。完成初始化后面板自动消失,不打扰老用户。

**FR-134 步骤状态自适应** —— 引导面板按当前状态显示「下一步该做什么」,检测三件事:① 是否有 OPEN 周期 ② 是否有用户自建账户(非内置模板)③ 本期是否已填报。据此把已完成的步骤打勾 / 灰掉,高亮下一步;三者都满足 → 面板隐藏。

**FR-135 「周期」概念一句话说明** —— 在引导面板和 `/entry` 顶部,用一句话 +(可选)极简流程图解释核心生命周期:**开周期 → 各成员填本期余额 → 关周期 → 出净资产 / 收益报告**。消除「周期是什么、为什么要开」的困惑。

## 9.4 关键决策(待评审,先给倾向)

- **形态 = 落地页内联引导面板(空状态)**,非弹窗向导、非独立 setup 页。理由:轻、不打断、完成即隐、移动端原生友好、零额外路由;符合现有「管理页热改」的克制风格。
- **判定靠现成查询**(有无 OPEN 周期 / 有无自建账户 / 本期是否填报),不引入新状态字段、不改 schema。
- 文案面向非技术家庭成员:不用「实例 / 初始化 / OPEN」等技术词(承 [[feedback_user_friendly_naming]]);图标用 inline SVG 不用 emoji(承 [[feedback_no_emoji]])。

## 9.5 红线 / backward-compat

- 纯增量:不改 schema、不动周期/账户/填报既有逻辑;对已有数据的存量家庭**面板不显示**(零打扰、零影响)。
- 判定查询要轻(落地页每次渲染都跑),走现有 service / 缓存,别拖慢首页。

## 9.6 验收基线

1. 全新部署 → 首次登录 → 落地页见引导面板 + 一句话流程说明 + 起步步骤直达按钮。
2. 开了周期 / 加了账户 / 填了报 → 对应步骤逐步打勾,三者齐 → 面板消失。
3. 有真实数据的家庭(存量)登录 → **不显示**面板。
4. `/entry` 顶部有「周期生命周期」一句话说明。
5. 移动端面板正常、不挡内容;无 emoji;文案无技术黑话。
6. mvn test 全绿 + qa-run 加守护(空状态面板存在 + 判定逻辑 + 有数据时不显示)。

## 9.7 分期

- **P1**:落地页空状态引导面板(三步判定 + 直达按钮 + 一句话流程)。
- **P2**:`/entry` 顶部周期概念说明 + 极简流程图解。
- 可一次性做完。
