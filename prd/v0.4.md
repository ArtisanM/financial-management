## 1. 版本目标

### 1.1 中产家庭的「摸清 5 问」

家庭账房的核心是回答 5 个问题。前 4 问 v0.1-v0.3 已基本答完,第 5 问完全缺失,**v0.4 主线就是补完第 5 问 + 把前 4 问的回答更聚焦(砍重复/流水)**:

| 5 问 | 服务定位 | 当前状态 | v0.4 行动 |
|---|---|---|---|
| 1. 我家有多少钱? | 摸清 · 快照 | ✅ v0.1 net worth | 保持 |
| 2. 钱分在哪? | 摸清 · 切片 | ✅ v0.1 配置环形 + 成员饼 + 账户分布 | 去 3 处重复显示 |
| 3. 钱怎么涨/跌的? | 摸清 · 时序 | ✅ v0.1 趋势线 / v0.2 本金 vs 收益分解 | 保持(核心图) |
| 4. 风险有多大? | 摸清 · 诊断 | ✅ v0.2 风险敞口 + 流动性 + 体检规则 | 去重复(图只在一处) |
| 5. **跑赢通胀和市场了吗?** | 摸清 · 对照 | ❌ **完全缺失** | **v0.4 新答** |
| (额外)**该怎么调?** | 增值不贬值 · 决策 | ⚠️ v0.2 体检只说"偏了" | **v0.4 给具体数字** |

### 1.3 v0.4 不做什么(非范围 · 进 v0.5 排队)

| 候选 | 砍掉理由 |
|---|---|
| **录入革命**(支付宝/微信 CSV → AI 归类)| 摸清这条 v0.3 已基本到位 · 等 v0.4 决策辅助建好,再降录入成本 ROI 更高;直接做现在=工具感,价值感弱 |
| **保险账户类型** | 跟"录入革命"配套做更顺(每年 1-2 次保费录入)· v0.5 一起 |
| **税务专项附加扣除** | 政策易变 + 跟官方个税 APP 重复 · 12 月一次性 · 维护负担重 |
| **多场景账本(AA / 装修 / 旅行)** | 与"家庭账房"硬约束冲突 · v0.1 起明确非目标 |
| **二级预算 / 消费品类细化** | 与"10 分钟/月"硬约束冲突 · v0.1 起明确非目标 |
| **新增账户类型** | 现有 6 类(CASH/STOCK/WEALTH/PROPERTY/LOAN/OTHER)够用 · v0.3 已扩展持仓三态(AUTO/MANUAL/CASH) |
| **失业 N 月情景模拟** | 价值中 · 需要更完整数据基线(保险账户)· v0.5+ |

### 主线 B · 摸清第 5 问 · 跑赢通胀和市场

#### FR-61a · 净资产趋势 + CPI 实际购买力对照 · P0

> **定位**:中产最痛"我以为我赚了,实际被通胀吃掉 N%"。dashboard 净资产趋势叠一条 CPI 折算后的"实际购买力"线 → 一眼看见名义 vs 实际差距。

**模型**:
- 双线展示:
  - **名义净资产**(实线 brass-deep,当前 v0.1 行为)
  - ⭐ **实际购买力**(虚线 forest,基期为锚 deflate · 公式:`real_t = nominal_t / (1 + cpi)^(t-t0)`)
- CPI 默认 **中国 2.0%**(近 5 年央行口径均值)
- 用户可改:dashboard 顶部右上角下拉 `[CPI 2%▼]` 切换 `2% / 3% / 5%`
- 选择持久化到 `family.cpi_assumption`(新字段)

**UI**:
- 图例增 "名义" + "实际购买力(CPI N%)"
- 末点 tooltip 显示 "名义 ¥X · 实际 ¥Y(折损 Z%)"
- 颜色:名义 = 品牌色 brass-deep · 实际 = 森林绿 forest 虚线

**验收**:
- [ ] dashboard 净资产趋势图含 2 条线 + 图例
- [ ] 切换 CPI 假设 → 实际线立即重算
- [ ] tooltip 显示名义 vs 实际差额 + 折损%
- [ ] family.cpi_assumption 持久化

#### FR-61b · 账户级 XIRR vs 基准 · P0

> **定位**:reports 账户级收益表每行加"基准"列 + 跑赢/输 pill。**复用 v0.2 早就有的 `product_category.benchmark_pct` 字段(沪深 300 / 标普 500 / 中债)**,这块弹药我们一直没用起来。

**实现**:
- 每个 account 查它的 product_category_code → benchmark_pct(可能 NULL)
- 计算:`diff = account_xirr - benchmark_pct`
- pill 染色:
  - `diff > 2%` → 🟢 跑赢 +X%(forest)
  - `-2% ≤ diff ≤ 2%` → ⚫ 持平(ink-soft)
  - `diff < -2%` → 🔴 跑输 -X%(rust)
  - 未设类目 / benchmark NULL → "—"

**UI**:
- 账户级收益表加列:`基准 % | 跑赢/输`
- 表头加说明:"基准来自产品类目,可在 /admin/product-categories 微调"

**验收**:
- [ ] 每个有 benchmark_pct 的账户行显示 基准 % 和 diff pill
- [ ] 未设类目的账户显示 "—"
- [ ] pill 颜色正确(forest/ink-soft/rust)

#### FR-61c · 家庭整体跑赢/输基准 · P0

> **定位**:reports 顶部 4 KPI 区(家庭 XIRR / TWR / ...)加一行"家庭基准 = N% · 跑赢 / 输 M%"。

**算法**:
- 家庭基准 = Σ(account.endBalanceBase × product_category.benchmark_pct) / Σ(account.endBalanceBase)
- 未设类目的账户用 v0.1 默认值(STOCK→6% / WEALTH→3% / CASH→0.5% / PROPERTY→3% / LOAN→负数 mortgage_rate)
- 家庭 XIRR - 家庭基准 → 跑赢/输

**UI**:

#### FR-62c · 应急金"不闲置"提示 · P0

> **定位**:复用 v0.3.3 流动性精细化数据,自动提示用户活期/工资卡有"超额闲置"应转货币基金。

**算法**:
- 当前 LIQUID 类账户余额合计(已 v0.3.3 含货币基金)
- 紧急储备需求 = `avg_monthly_expense × 应急倍数`(默认 6 个月,可改 · 复用 FR-50c 应急目标设定值)
- 阈值:`LIQUID 余额 > 应急需求 × 1.5x`(超 1.5 倍 = 闲置)
- 超额 X = `LIQUID - 应急需求`

**UI**:
- **dashboard 紧急储备 KPI 旁**:有超额时显黄色提示 banner:
  > 💡 应急金充足 + 超额 ¥X · 建议转货币基金(余额宝 / 零钱通)以获取 2%+ 年化
- 链接到 `/admin/product-categories#MONEY_FUND` 查看货币基金参考产品

#### FR-62d · 提前还贷 vs 投资 NPV 决策器 · P0

> **定位**:新页 `/reports/refinance` · 用户输入剩余房贷情况,系统算 "现在多还 ¥N 给房贷 vs 投资 ¥N" 的 NPV 对照,给推荐。

**输入**(form):
- 剩余房贷本金(默认从 LOAN 账户拉)
- 房贷年利率(默认从 family.mortgage_rate 取 / 用户改)
- 剩余年限(默认从房贷账户计算)
- 准备多还 ¥N(用户输入)
- 投资预期年化(默认 = 家庭 XIRR · 用户可改)

**算法**:
- 提前还贷收益 = `N × ((1+r_loan)^n - 1) / r_loan`(节省利息)
- 投资收益 = `N × ((1+r_invest)^n - 1)`(复利增值)
- diff > 0 → 推荐投资 · diff < 0 → 推荐还贷
- 边界:r_loan > r_invest → 必还(节省 = 确定 vs 投资 = 不确定)

**UI**:
- 表单(5 字段)
- 算完显双柱(还贷收益 vs 投资收益)+ 推荐
- AI 解释(可选,复用 LlmOrchestrator)

**验收**:
- [ ] /reports/refinance 200 + 表单完整
- [ ] 提交后显双柱 + 推荐
- [ ] 边界(r_loan > r_invest)推荐必还
- [ ] 默认值正确(从 LOAN 账户 + 家庭 XIRR 自动拉)

## 12. v0.4.4-hotfix · prod 部署应急(2026-05-14)

### 12.1 触发

用户把 v0.4.3 + v0.4.4 一并发布到 prod 后 `/dashboard` 直接挂,Chrome 报 `ERR_INCOMPLETE_CHUNKED_ENCODING`。

### 12.2 真因(从 prod 日志锁定 · 不是猜测)

时间戳最早的 ERROR(14:19:15):

```
TemplateProcessingException: Could not parse as expression:
  "${liquidSurplus.emergencyMonths} + ' 月 · ¥' + #numbers.formatInteger(...)"
  (template: "dashboard/_region" - line 157, col 13)
```

L157 + L158 两行表达式语法错:`#numbers.xxx()` 这类 utility 调用**必须在 `${...}` 内部**,代码却写成了 `${var} + 'str' + #numbers.xxx(...)`(`#numbers` 在 `${...}` 外)。

**为什么 beta 不挂 prod 挂**:L148 的 `th:if` 只在「LIQUID > 应急需求 × 1.5x」时渲染 banner。beta 数据应急金没超额 → section 跳过 → 错误表达式从不被 parser 触碰。prod 应急金确实超额 → banner 进入 → parse 立即炸。

**为什么 Chrome 看到 INCOMPLETE_CHUNKED**:Thymeleaf 默认 chunked streaming · head fragment 渲染完成 + flush → response 已 commit · Spring 无法 forward 到 /error · 连接直接掐断。

### 12.3 修复链(3 commit · 按时间倒序)

| Commit | 角色 | 改动 |
|---|---|---|
| `69ce5b6` | **真因** | dashboard/_region L157+L158 表达式包到一对 `${...}` 内 |
| `9218442` | 兜底 | fragments/nav.html 两个「退出」form + dashboard/_region CPI 切换器 form 加 `th:if="${_csrf != null}"` |
| `87e644e` | 兜底 | fragments/layout.html head meta `${_csrf.token}` / `${_csrf.headerName}` 改 null-safe |

兜底 commit 让以后任何模板挂时 /error 自己能正常渲染显示给用户(不会被 _csrf null 二次炸)。

### 12.4 诊断教训

- 看 prod stack 时,**第一条 ERROR(时间戳最早)才是 root cause**。我先看到 14:19:16 的 `_csrf.token` null 错(forward 到 /error 的二次炸),自动假设是 root cause,先打了两个 _csrf null-safe hotfix —— 应该按时间戳找最早那条
- 不要拿到栈底的 `Caused by:` 就停 · 而是顺着时间戳往上找第一个 ERROR · 那一条的 message 通常含 `template + line + 表达式` 三要素
- Thymeleaf 表达式语法纪律:**`#numbers.xxx()` / `#temporals.xxx()` 等 utility 调用必须完整地在 `${...}` 内部**。错例 `${var} + 'str' + #numbers.xxx()` 看似合法但 Thymeleaf parser 拒绝(因为它把 `${...}` 外的 `+ #numbers...` 当作非法的字符串拼接)

## 19. v0.4.11 · prompt 占比 bug 修复 + 严禁 LLM 算术(2026-05-14)

### 19.1 触发

用户反馈 LLM 输出胡说:「股票类仅占 3.4%(¥376 万 / ¥1095 万)· 远低于退休目标」。
但实际比例完全不对(¥3,762,836 / ¥10,950,000 ≈ 34%,不是 3.4%)。
用户的更深刻反思:**不应该让 LLM 做任何数学计算 · 所有计算类指标应该工程算好填给 LLM**。

### 19.2 真因 · 两层

**层 1 · `pct1(ratio)` 没 ×100**(基础 bug)

```java
// PromptBuilder L137(全家诊断 资产配置段)
sb.append(" · 占比 ").append(pct1(s.ratio())).append('\n');

// pct1 实现
private static String pct1(BigDecimal v) {
    return v.setScale(1, RoundingMode.HALF_EVEN).toPlainString() + "%";
}
```

`AllocationSlice.ratio()` 返 `0.442`(小数比例),`pct1` 直接显「0.4%」而不是「44.2%」。

`RiskBucket.ratio()`(L147)+ `AccountSummary` 占总资产比例(L223 反向 bug:`benchmarkPct` 是 `8.00` 已是百分比形式,multiply(100) 变 800,显「800%」)同款问题。

prompt 给 LLM 的:
```
- 现金 (CASH): ¥95710 · 占比 0.0%     ← 错(实际 2.4%)
- 股票 (STOCK): ¥1779269 · 占比 0.4%   ← 错(实际 44.2%)
```

LLM 拿到错误数据自然推理出"权益占比严重不足"。

**层 2 · LLM 不该做任何算术**(根本)

即使数字对了 · LLM 也会:
- 二次推算占比(经常算错 0.5pp 这种)
- 自己组合「¥X / ¥Y = Z%」(几乎必错)
- 凭"印象"造数字

→ Prompt 必须**禁 LLM 做四则运算** · 让它只负责"语言表达 + 跨规则综合判断"。

### 19.3 修复

| 改动 | 目的 |
|---|---|
| 新增 `pctFromRatio(BigDecimal ratio)` · 自动 ×100 | 命名清晰区分 · 防 caller 弄错 |
| `pct1(s.ratio())` → `pctFromRatio(s.ratio())` (allocation + risk) | 修 0.4% bug |
| `pct1(benchmarkPct.multiply(100))` → `pct1(benchmarkPct)` (account diagnose L223) | 修 800% bug |
| SYSTEM_DIAGNOSE 加「最高优先级:数字严禁原则」5 条 | 禁四则运算 · 强制照抄 |
| userPromptForFamily 顶部标注「⚠ 重要 · 以下所有数字均由系统预先计算 · 你只能引用 · 不要自己运算」 | 强化约束 |

## 20. v0.4.14 · 填报规范化 + DDL 强提醒(FR-63 · 2026-05-18)

### 20.1 触发

家庭记账靠"每月找一天 10 分钟全家填完",但没规范"何时填什么" → 成员忘填 / XIRR 时间精度差。用户要求:① 定义填报模板让全家有统一节奏 ② 填报页给推荐方案提示 ③ 截止前强提醒(短信兜底)④ 短信需要设置页(aksk + 成员手机号)· 且**手机号 / aksk 属私密信息,绝不可出现在任何 LLM 交互中**。

### 20.2 FR-63a · 家庭级填报模板(全家统一)

| code | 名称 | 节奏建议 |
|---|---|---|
| T1_REALTIME_INCOME_MONTHEND_EXPENSE(默认) | 实时收入 · 月末支出 | 收入随发生填(发薪日更新收入+余额);支出月末倒数 2 天集中填 |
| T2_MONTHEND_BATCH | 月末一次清 | 全部收入/支出/余额月末倒数 2-3 天一次性填完 · 最省心 |
| T3_WEEKLY_ROLLING | 每周滚动 | 每周日复盘填本周收支 · 适合流水频繁 |

- 模板**只规范节奏建议 + 填报页提示文案**,不改 period_type 机制;填报截止统一是 `period.periodEnd`。
- 家庭级(非成员级)· 默认 T1 · 老 family 无字段自动落 T1。

### 20.3 FR-63b · 填报页推荐方案提示

`/entry` 在视图切换区下方插提示 banner:显示当前模板名 + `hintText` + 距 `period.periodEnd` 天数。距截止 ≤ 2 天或当天 → 红色强样式;已过截止 → "本期已过截止日";仅 OPEN 周期显示。

### 20.4 FR-63c · 截止前强提醒调度 + 文案

**调度**

- 截止时间 = `period.periodEnd`;`family.report_remind_lead_days`(默认 2)= 截止前几天进入提醒窗口。
- 调度 `@Scheduled(cron="0 0 10,20 * * *", zone=Asia/Shanghai)`(每天 10:00 / 20:00)。
- 渠道抽象 `NotificationChannel`(可插拔):**短信为主**(默认 provider = **阿里云** `SmsAliyunChannel`)**+ 站内 banner 强化兜底**(`InAppBannerChannel` · 永远可用)。
- 去重:`report_reminder_log` UNIQUE(family,period,member,channel,date) + `INSERT IGNORE` → 同成员同渠道当天最多 1 次(cron 一天 2 次仅作早失败晚补的容错)。
- 只对处于提醒窗口、且**本期未提交完成**的成员触达。

**短信 provider 选型**(详见 [`tech-design/v0.4.md`](../tech-design/v0.4.md) §16.9 决策 35):
- **MVP 选 阿里云**(`dysmsapi.aliyuncs.com` · RPC HMAC-SHA1)。**user-facing 决策维度**:
  - 充值门槛低:**无最低充值**(按量付费直接扣账户余额)/ 套餐包 **6.9 元 / 200 条**起步(0.034 元/条);云片硬性 ≥ 50 元起步,对家庭单户用量(全年 ≈ 100 条)过高。
  - 用户已有账号(Java 工程师标配 · 免再注册新平台 + 实名)。
  - 大厂稳定 + 国际短信成熟(未来扩展空间)。
- DB `family_notify_config.sms_provider` 字段预留多 provider 切换(MVP 实现 `aliyun`;接口设计可加 `yunpian` / `tencent` 等,只需新写一个 Channel 实现)。
- ⚠ **个人资质阻塞**(2026 阿里云官方原文:"个人账号的自用资质无法通过签名实名制报备" · 见 [Aliyun getting-started 页](https://help.aliyun.com/zh/sms/getting-started/get-started-with-sms)):**必须企业实名账号**才能拿到签名。**个人账号 → 短信渠道走不通,但站内 banner 永远可用**(`InAppBannerChannel` 不依赖任何外部凭证 · §20.6 失败降级 · 核心提醒能力不塌)。

**文案 · 两套(因为渠道字数约束差异巨大)**

阿里云短信模板需运营商**实名预报备**(无法运行时改文案 · 占位语法 `${var}`),
正文 ≤ 67 汉字(含 `【签名】` 前缀)、单变量值 ≤ 20 字符;所以**短信走"在限制内塞尽量多变量"**,
**站内 banner 不受限走富信息**。两者承载的信息维度对齐:

(1)**短信模板**(报备时定死格式 · 运行时把变量填进 `TemplateParam` JSON 调 dysmsapi `SendSms`):

```
【家庭账房】${brand}账本 ${period} 提醒:距记账截止${days}天,
家庭已完成 ${progress},请尽快填报。
```

变量:
| 变量 | 含义 | 示例 | 长度上限 |
|---|---|---|---|
| `${brand}` | 家庭别名 = `family.brandText`(空时 fallback 家庭名) | 狗窝 | ≤ 8 |
| `${period}` | 本期标识 | 2026年5月 / 第20周 | ≤ 10 |
| `${days}` | 距截止剩余天数 · 0 = 今天截止 | 2 / 0 | ≤ 2 |
| `${progress}` | 全家填报进度 | 2/4人 | ≤ 8 |

(2)**站内 banner**(`/entry` 页 · 见 §20.3 · 富信息无限制),分三栏:
- 左:推荐填报方案名 + `hintText` 节奏建议 · 引导用户"今天应该填什么"
- 中:本期标识 + 截止日期(`MM 月 dd 日`)+ 家庭进度 `N/M 人已填` + 我自己已填/未填的徽标
- 右:距截止天数 · 状态 pill —— `>2 天` 中性 / `0-2 天` 红色强样式 / `<0` 显「已过截止」

**为什么不把短信做更长**:多变量对应阿里云模板审核更严(可能反复打回);本方案 4 变量是
"信息量↔报备通过率↔成本"三者平衡点。后续若有需要可单独立项做"长短信"或"群发批量"。

#### 20.5.1 ⊕ 一键测试短信(用户反馈追加)

**目的**:管理员配完 aksk + 签名 + 模板后,在本页**立刻**验证"这套配置真的能发出短信",
不必等真实截止窗口才发现配错。

**交互**:

- 配置区下方加按钮「▷ 发送测试短信」+ 目标手机号选择器:
  - 默认从已配手机号的成员列表里选(下拉);
  - 也允许输入一个临时手机号(11 位)做一次性测试(不落库)。
- 点击 → 后端 `POST /admin/reminders/sms-test`:
  1. 读当前 family 的 `family_notify_config` · 任一项(aksk/签名/模板)缺 → 立刻返回"配置不完整",**不调阿里云**。
  2. 用**测试专用变量值**渲染模板:`brand=<brandText>` / `period=配置测试` / `days=99` /
     `progress=测试模式`(让运营商日志一眼能区分测试 vs 真实提醒)。
  3. 调 `SmsAliyunChannel.send(...)` 发出 · 拿阿里云返回的 `Code + BizId + RequestId`。
  4. 审计:**走 `audit_log`(非 report_reminder_log)** —— 详见 [tech-design §16.10](../tech-design/v0.4.md):
     report_reminder_log 的 UNIQUE 当天去重为日常调度服务,与 SMS_TEST「每分钟可点 3 次」冲突;
     SMS_TEST 本就是"管理员主动行为审计",语义归属 audit_log 更对。detail 不含手机号 / aksk 明文。
  5. 前端 toast:成功显示「✓ 测试短信已发送 · 请查收(BizId: ...)」,
     失败显示「✗ 阿里云返回 Code=isv.xxx」(配置错误码原样吐给管理员去查阿里云文档)。
- **限流**:每位管理员每分钟最多 3 次测试,防误点 / 防刷短信成本。

**为什么单独做 sms-test 不复用 ⑤ 手动触发调度**:
- ⑤ 是"在真实窗口下走完整链路"(只对未填成员发 · 真截止天数 · 当天去重),在非窗口期(还差 10 天截止)就是空跑,验证不了短信。
- sms-test 是"绕过窗口 / 去重直接试一发",和真实提醒严格隔离(`channel=SMS_TEST` 不参与日常去重)。

**测试场景变量值**(用真实账期 + 测试占位 days):
- `brand=<family.brandText>` 真实
- `period=<period.periodEnd.yyyy-MM-dd>` 真实账期截止日 · 过 Aliyun「时间」校验
- `days=-1` 测试标识(运营商日志一眼可辨"非真实窗口")· 过「金额/数量」校验(允许负数)
- `progress=100` 测试满

未配 OPEN 周期时拒绝测试(`flashError`),引导用户先去『管理 → 周期』开期。
