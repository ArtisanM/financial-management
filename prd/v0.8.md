# PRD · v0.8 · 自用两月后的体验/指标补课 + 时间筛选器按账期大改造

> 本版由作者自用两个月后的真实痛点驱动:① 移动端关键指标看不到 ② 时间筛选器形同虚设 ③ 转账对收益率的影响 ④ 账户「近况」图是假的;并据此把**时间筛选器**从「随便选个区间但啥也不改」重做成**以账期(月)为一等公民**的真筛选,加 MoM/YoY 跨期对比;再补**列表按列排序**。
>
> 诊断结论均已核对真实代码(带 file:line),见各 FR 的「现状」。

## 0. 背景

数据底座是**按月度账期(period)结算的账户快照**(period_snapshot 等),不是逐笔流水。两个月自用暴露:筛选器、移动端、近况图等没跟上这个「按月」的本质,部分指标要么看不到、要么不随筛选变、要么是假数据。本版回归「按账期」这条主线把它们做对、做真。

## 1. 目标用户 & 场景

自托管的家庭主用户,**手机端**每月填报 + 平时随手看;会想「这个月比上月/去年同月怎么样」「近半年净资产怎么走」「这个账户收益率排第几」。

## 2. 非目标(本版不做)

- 不下钻到日级/逐笔(仍是月度颗粒度;股票日级行情是估值用,不改账期模型)。
- 不做任意自定义日期区间(`2026-03-15 ~ 2026-05-20` 这种)——既然只有月度数据,筛选只对齐到**账期**。
- 不做指标预计算落库的离线管线(理由见决策 92)。
- 配置/成员/账户分布图的「区间首末对比」本版先不做(推后)。

### B 组 · 时间筛选器按账期大改造(本版核心 · Q2)

- **FR-144 · 筛选以「账期(月)」为颗粒度,不再是模糊区间**
  - 现状:`range`(1M/3M/6M/YTD/1Y/ALL)只真正驱动了「净资产趋势图 x 轴」+「年化 TWR / 账户 XIRR 的窗口重算(显示成小字)」;头部 5 个 KPI 有 4 个恒为最新期、储蓄率/月均收支完全绕过筛选器、3 个分布图恒为最新期;且选 `1M` 时净资产 Δ 变「—」(`DashboardController.java:273`)。底层**有能力**按任意月窗口算(FactSlice 带 rangeStart/rangeEnd,XIRR/TWR 是纯函数)——是「只接了图、没接 KPI」。
  - 要求:筛选模型重做为两个正交控件(**待评审,见决策 91**):
    1. **观察账期(as-of period)**:默认最新一期,可选任意历史月。所有**点状指标**(净资产/总资产/总负债/资产配置/账户当前价值/账户列表)都按所选这一期呈现 —— 即「回看某个月的全家状态」。
    2. **窗口(近 N 期 / 今年以来 / 全部)**:对齐到账期,驱动**趋势图 x 轴** + **窗口型指标**(区间净资产变化、区间 XIRR/TWR、窗口化储蓄率/月均收支)显式、显眼地展示(不再是角落小字)。
  - 修掉 `1M`(单期)时净资产 Δ 变「—」的 bug:单期时 Δ 退化为「MoM」语义(见 FR-145)而非空。
- **FR-145 · 跨期对比指标 MoM / YoY**
  - 在以账期为颗粒度的前提下,头部关键 KPI(净资产、本期投资收益、收益率、储蓄率等)附**环比 MoM**(对所选 as-of 期 vs 上一账期)与**同比 YoY**(vs 去年同月账期)。
  - 找不到对应对比账期(数据不足 1 年 / 中间有缺月)→ 该项显「—」或「数据不足」,不臆造。作者生产现仅 1~2 期 → MoM 可能有、YoY 必然「数据不足」,要如实。
  - **计算结果是否落库:不落库,实时算**(决策 92)。
- **FR-146 · 筛选状态可分享/可回链**:as-of 期 + 窗口编码进 URL query(承现有 `?range=` 习惯),刷新/分享保持;HTMX 局部刷新沿用现状。

### C 组 · 跨币种转账正确性(Q3)

- **FR-147 · 转账金额支持「转入账户币种」**
  - 现状:**同币种转账下收益率不被污染是正确的**——转账在转出/转入账户各自的 XIRR 现金流里被正确当作外部现金流计入(`FactMapper.xml` 的 `transfer_in_orig/out_orig` → `xirrForAccountRows`),PnL 恒等式也扣掉了转账。**唯一问题是跨币种**:`transfer` 表只有一个无币种的 `amount`(`V1__init.sql:161-179`),CNY→USD 转账时转入端用错币种的数 → 该账户 PnL/XIRR 被污染、家庭级不再净为零。

- **FR-151 · 计算口径与场景正确性(对应你问的 #4)**
  - **总则**:每账户每期 `Δ期末余额` 拆 = 外部流入(income−expense,源 `cash_flow`)+ 转账(transferIn−transferOut,源 `transfer`)+ 投资损益 PnL(余项=估值变动);所有收益指标据此派生(原币种算 PnL,再 ×当期汇率得本位币)。
  - **逐指标公式**:累计损益=ΣPnL;本金=Σ(income−expense+transferIn−transferOut);XIRR=以各期净外部流入为现金流+首期末余额为初值+末期末余额为终值解,<12月退累计;本期收益率=本期PnL÷期初;TWR=各期(期末−净流入)/期初连乘−1;占比=账户现值÷家庭净资产;MoM=本期vs上一账期、YoY=vs去年同月账期,缺对比期显「数据不足」。
  - **场景归类**(已核实,见对话内总则表):股票自动估值→正确归 PnL;转账→正确剔除(跨币种除外=FR-147);手动改余额未记流水→归 PnL(模型假设,需 UI 提示用户记流水);草稿不计、归档 dashboard 不计、reopen 实时重算、中途新增账户 XIRR 从首个有快照期起。
  - **本版要决策的三个正确性问题**(见决策 97):① 跨币种转账(FR-147 已含)② 手动改股票内部现金行被当 PnL 污染收益率(Problem B)③ 本位币收益率混入 FX 汇率损益、未单列(Problem C)。Problem A(现金联动只动账户内部现金行、不联动独立现金账户)→ 倾向文档澄清,不改模型。

- **FR-152 · 预实分析(预期收益 vs 实际收益)**
  - **预期收益率来源**:账户所属品类的 `product_category.benchmark_pct`(已存在,admin 可微调)为默认;新增**可选的账户级覆盖** `account.expected_return_pct`(可空,NULL → 回落品类默认)——让用户对单个账户设自己的预期(决策 98)。
  - **对比维度**:① **收益率**:实际(年化 XIRR / 区间收益率)vs 预期% → 差异(bps)+ 达成率(实际/预期)+ 跑赢/跑输标识(复用 `BenchmarkComparator`,只换「基准」为「预期」措辞,合并展示);② **金额**:预期收益金额(≈ 平均投入 × 预期% × 持有年数)vs 实际累计损益(cumPnl)→ 差额。
  - 数据不足(无 benchmark 也无 override / 持有 <2 期)→ 显「未设预期」/「数据不足」,不臆造。
  - 与可配置指标集打通:「预实分析」作为账户级指标可勾选(默认 ✓),在列表/卡片显「实际 vs 预期」紧凑形态,详情看金额维度。

## 4. 关键决策(**待评审 · 先给倾向**)

- **决策 91 · 筛选器交互模型 = 「as-of 账期(点状) + 窗口(趋势/窗口指标) + MoM/YoY(对比)」三件套**
  - 倾向:把「看某个月的状态」和「看一段时间的走势/回报」拆成两个正交控件,而不是塞进一个 1M/3M/… 单选。理由:这俩是不同问题,旧设计把它们混在一个 range 里,导致点状 KPI 无所适从(只能恒取最新期)。拆开后每个 KPI 该响应谁很清楚。
  - 备选(否):维持单一 range 单选,只把更多 KPI 接上 range。更小改动,但语义仍别扭(选 3M 时「净资产」到底显哪一期?)。
  - 备选(否):纯单期浏览(只选月,无窗口)。丢掉趋势/区间回报的价值。
  - **需要你拍板**:接受「双控件 + MoM/YoY」模型?还是想要更简的形态?(我可出 preview 静态原型再定。)
- **决策 92 · MoM/YoY 及窗口指标 = 实时算,不落库**(你点名让我想的)
  - 选定:**不落库,按 FactSlice 实时算**。
  - 理由:① MoM/YoY/区间回报都是**已存月度快照的纯函数派生**,数据量极小(一个家庭几十~一百多行 period),每次 dashboard 算一遍成本可忽略;② 本项目架构本就是 factview「大宽表实时派生」+ calc 纯函数,落库会破坏这层干净分离;③ **账期可被 reopen/改写**(`period_reopen_log` 存在),落库的派生指标会变陈旧、要额外失效/重算管线,得不偿失;④ 已关账报表的「快照」诉求已由 v0.5.6「报表已关账快照」覆盖,属另一层(冻结一次性叙述),不是 dashboard 实时指标。
  - 备选(否):建 `period_metric` 派生表、关账时算好落库。只有当计算贵或要离线分析才值得;此处不满足,且引入一致性负担。
  - 一句话:**原始月度快照落库(已是),派生指标一律实时算。**
- **决策 93 · 跨币种转账 = 加可空 `to_amount` 列,旧数据落 NULL**
  - 选定:`transfer` 加 `to_amount DECIMAL NULL`(NULL=同币种,用 `amount`);`FactMapper.xml` 转入额改 `SUM(COALESCE(to_amount, amount))`,转出额仍用 `amount`(转出账户本币)。**计算逻辑零改动**,改动只在 schema + 数据抽取层 + 转账录入/编辑表单(跨币种时多填一个到账金额)。
  - backward-compat:线上历史 transfer 全部同币种 → `to_amount` 默认 NULL → `COALESCE` 回落 `amount`,旧记录零影响、零回填。承 [[feedback_prod_backward_compat]]。
  - 备选(否):转账存 base 币种金额 + 两端按汇率换算。要引汇率快照、更复杂,且对同币种(99% 场景)是过度设计。
- **决策 94 · 列表排序 = 客户端 JS 三态,默认不动后端**
  - 倾向:数据已在 DOM 且量极小,纯前端 JS 排序(三态切换 + `aria-sort`),无后端往返、无新 endpoint。
  - 备选(否):HTMX 服务端排序(加 `?sort=col&dir=asc`)。数据小,没必要;但若将来分页再说。
- **决策 95 · 可配置指标集 = dashboard 与 reports 共用一套勾选,各页按上限裁剪**(回答你的 #1)
  - 选定:一套「我关心的指标」勾选(按 家庭级/账户级 分组),dashboard 与 reports **共用**;每页按自己的展示上限(决策 96)取前 N(按固定优先级)。
  - 理由:指标定义同一套,集中一处管 = 认知负担最小;两页诉求不同靠「上限+优先级」而非两套配置来调和。dashboard 装不下的,reports 看全、或进账户体检看全集。
  - 备选(否):每页各一套独立配置。更灵活但 2× 配置 UI + 用户易困惑;v1 不做,留「将来可加 per-页覆盖」。
- **决策 96 · 展示上限**(回答你的 #3)
  - 手机 dashboard 账户**卡片首屏 ≤ 4 个指标** + 账户名 + sparkline,其余点开看;手机家庭 KPI **≤ 4 张卡(2×2)**。
  - 桌面 dashboard 账户表 **≤ 6 列**;桌面家庭 KPI **≤ 6 张**。
  - reports(桌面为主)**不设上限**,展示全部已勾选;账户体检页恒为**全集**。
  - 勾选数超上限 → 该页按固定优先级取前 N,并提示「更多见报表/体检」。优先级 = 目录表的默认/排序顺序。
- **决策 97 · 三个计算正确性问题的处理**(回答你的 #4 的延伸)
  - **跨币种转账**:加 `to_amount`(决策 93,FR-147)。
  - **手动改股票内部现金行污染 PnL(Problem B)**:**✓ 本版纳入(已确认 2026-06-23)**。`updateCashAmount` 这类「主动调现金」记一笔对应的 income/expense(或视为账户内部 cash 调整不计入 PnL),从投资损益中剔除。tech-design 定具体落点。
  - **FX 汇率损益未单列(Problem C)**:**✓ 本版纳入(已确认 2026-06-23)**。本位币收益率**保留含 FX**(对家庭是真实损益),对非本位币账户**额外提供「原币收益率」**让用户看到剔除汇率后的标的表现;不强行拆分家庭级 PnL。
  - **Problem A(现金联动 vs 独立现金账户)**:文档澄清「现金联动只动股票账户内部现金行,不自动扣独立银行账户;跨账户出资请记一笔转账」,不改模型。
- **决策 98 · 预期收益来源 = 品类 benchmark_pct 默认 + 可选账户级覆盖**(回答你的「预实分析」)
  - 选定:默认用账户所属品类的 `product_category.benchmark_pct`(已存在);新增可空列 `account.expected_return_pct`,设了用它、没设(NULL)回落品类默认。**复用 `BenchmarkComparator` 算实际 vs 预期**,只是把「基准对照」与「预实分析」合并为同一展示。
  - backward-compat:`expected_return_pct` 默认 NULL → 旧账户全部回落品类 benchmark,零回填、零影响。
  - 备选(否):每账户强制设预期 → 录入负担;品类默认已能开箱即用。
